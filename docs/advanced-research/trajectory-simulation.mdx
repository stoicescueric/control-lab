# Trajectory Simulation

<Difficulty level="Hard" />

<Abstract>
Trajectory simulation is the capstone problem because it forces every earlier idea to work together: the robot must estimate where it is, plan where it should go, and follow that reference under real actuator and sensor limits. This page lays out the research architecture for validating that stack against FTC telemetry, turning the curriculum from separate lessons into one measurable autonomous system.
</Abstract>

<Admonition type="note" title="Research status">

This page is intentionally written as a **research plan**, not a finished paper. It defines the model, measurements, experiments, and success criteria that the final capstone should satisfy once on-robot telemetry is available.

</Admonition>

## The question

Every autonomous routine is a chain of approximations. Odometry approximates the robot pose. A motion profile approximates what the drivetrain can physically execute. A path follower approximates how to correct back to the reference while the robot is already moving. Feedforward approximates the voltage needed before feedback catches up.

The capstone question is:

> How accurately can a simulation predict the tracking error of an FTC autonomous path before the robot runs it?

That question is useful because it is falsifiable. If the simulator predicts a smooth path but telemetry shows large cross-track error, heading lag, or missed timing windows, the difference points to something real: wheel slip, wrong acceleration limits, incorrect feedforward, latency, localization drift, or a follower parameter that only works on a clean plot.

## The pipeline being simulated

The simulation should model the same pipeline the robot uses during autonomous:

1. **State estimation** - advance pose with dead-wheel odometry or drivetrain encoder odometry.
2. **Reference generation** - choose the path and speed profile the robot is trying to follow.
3. **Path following** - compute a steering or field-vector command from the estimated pose.
4. **Actuator command** - convert the desired motion into motor powers or voltages.
5. **Telemetry comparison** - compare simulated pose, measured pose, command history, and timing.

The key discipline is that the simulator should not use a cleaner controller than the robot. If the robot follows a path with pure pursuit, guided vector fields, or a Pedro-style follower, the simulator should run that same algorithm or a deliberately documented simplification.

## Minimal model

A useful first simulator does not need perfect physics. It needs the right failure modes.

The state can start as planar pose and velocity:

$$
\mathbf{x} =
\begin{bmatrix}
x \\
y \\
\theta \\
v_x \\
v_y \\
\omega
\end{bmatrix}
$$

where:

- $x$ and $y$ are field position.
- $\theta$ is robot heading.
- $v_x$ and $v_y$ are field-relative translational velocity.
- $\omega$ is angular velocity.

A simple discrete update is:

$$
\mathbf{x}_{k+1}
=
\mathbf{x}_k + f(\mathbf{x}_k, \mathbf{u}_k)\,\Delta t
$$

where:

- $k$ is the loop index.
- $\mathbf{u}_k$ is the drive command at that loop.
- $\Delta t$ is the measured loop period.
- $f$ is the motion model that converts the current state and command into state rate.

This is intentionally the same discretization pattern from [From Math to a Loop](../software-architecture/discretization.mdx). The simulator is not separate from the curriculum; it is the curriculum running end to end.

## What should be measured

The final capstone should report more than a pretty path overlay. At minimum, collect:

| Metric | What it reveals |
| --- | --- |
| Cross-track error | Whether the follower stays on the path |
| Heading error | Whether translation and rotation are tuned together |
| Loop time distribution | Whether the algorithm fits inside the Control Hub budget |
| Command saturation | Whether requested motion exceeds what the drivetrain can produce |
| Battery voltage | Whether feedforward and motor output changed during the run |
| End-state error | Whether the robot arrives ready for the next mechanism action |

The important plot is not only simulated path versus measured path. It is the residual:

$$
\mathbf{r}_k =
\mathbf{x}_{\text{measured},k}
-
\mathbf{x}_{\text{simulated},k}
$$

where $\mathbf{r}_k$ is the error between measured telemetry and simulation at loop $k$. A good simulator makes that residual small and explainable.

## Validation experiments

Use experiments that isolate one failure mode at a time:

1. **Straight-line acceleration** - estimates feedforward, acceleration limit, and voltage sag.
2. **Constant-radius turns** - exposes centripetal correction and lateral slip.
3. **S-curve path** - stresses heading control and changing curvature.
4. **Full autonomous segment** - tests interaction with mechanism timing and state-machine transitions.

Each run should export the same telemetry schema: timestamp, pose estimate, target pose or target path parameter, drive command, battery voltage, and loop time.

<Admonition type="warning" title="Hardware reality: simulation is not proof">

A simulator can explain and predict. It cannot certify the robot. Wheel scrub, carpet variation, battery voltage, mechanism impacts, and localization resets all create behavior that a simple model will miss unless telemetry forces the model to account for them.

</Admonition>

## How this becomes the final capstone

The finished research page should replace this plan with measured results:

- the exact robot model and units;
- the path follower under test;
- the telemetry format;
- simulation versus robot plots;
- residual error statistics;
- what changed after the model exposed a real robot issue.

Until those measurements exist, the honest academic contribution is the architecture: a reproducible method for connecting control theory to robot behavior, not a claim that the simulator is already validated.
