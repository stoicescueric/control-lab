# Simulating Physical Reality

<Difficulty level="Hard" />

<Abstract>
Accurate projectile control starts before the robot fires: the simulator must predict how the projectile flies and how much energy the launcher actually transfers. This lesson turns the research paper's physics stack into a three-step dashboard: numerical integration, aerodynamic drag, and contact-loss calibration.
</Abstract>

The practical failure is simple: a robot can spin the flywheel to a speed that looks correct, fire from the correct distance, and still miss because the model is too clean. Vacuum parabolas ignore drag. Euler integration accumulates error when the time step gets coarse. A flywheel's surface speed is not the same thing as projectile exit velocity.

That is why this module begins with a simulator instead of code. You can only compensate a shot in real time if the offline model already knows the important losses.

<PhysicalRealityDashboard />

## Step 1: Numerical integration is part of the model

Projectile motion with quadratic drag is not a one-line parabola. The acceleration depends on the velocity, and the velocity is changing during the flight:

$$
\dot{\mathbf{p}} = \mathbf{v}
$$

$$
\dot{\mathbf{v}} =
\begin{bmatrix}
0 \\
-g
\end{bmatrix}
-
k \|\mathbf{v}\| \mathbf{v}
$$

<EquationLegend
  items={[
    {symbol: '$\\mathbf{p}$', meaning: 'projectile position vector', unit: 'm'},
    {symbol: '$\\mathbf{v}$', meaning: 'projectile velocity vector', unit: 'm/s'},
    {symbol: '$g$', meaning: 'gravitational acceleration', unit: 'm/s^2'},
    {symbol: '$k$', meaning: 'lumped drag coefficient for the simplified demo model', unit: '1/m'},
    {symbol: '$\\|\\mathbf{v}\\|$', meaning: 'speed, the magnitude of the velocity vector', unit: 'm/s'},
  ]}
/>

Euler integration takes one slope and trusts it for the entire step:

$$
\mathbf{x}_{k+1} = \mathbf{x}_k + f(\mathbf{x}_k)\Delta t
$$

RK4 samples the slope four times inside the same step:

$$
\mathbf{x}_{k+1}
=
\mathbf{x}_k
+
{\Delta t \over 6}
\left(
\mathbf{k}_1 + 2\mathbf{k}_2 + 2\mathbf{k}_3 + \mathbf{k}_4
\right)
$$

The point of RK4 is not that it is magically "more advanced." It is that the acceleration changes during the time step, so the simulator should not pretend the first slope remains true until the next loop. In the dashboard, increasing $\Delta t$ makes Euler drift away first; RK4 stays usable longer because it samples the curvature of the state update.

## Step 2: Drag is not a small correction

For a light, blunt game element, drag can be the same order of magnitude as gravity at shooter speeds. The standard quadratic form is:

$$
\mathbf{F}_d = -{1 \over 2}\rho C_d A \|\mathbf{v}\|\mathbf{v}
$$

<EquationLegend
  items={[
    {symbol: '$\\mathbf{F}_d$', meaning: 'aerodynamic drag force, opposite the velocity', unit: 'N'},
    {symbol: '$\\rho$', meaning: 'air density', unit: 'kg/m^3'},
    {symbol: '$C_d$', meaning: 'drag coefficient; the demo uses 0.47 from the paper prompt', unit: 'unitless'},
    {symbol: '$A$', meaning: 'projectile frontal area', unit: 'm^2'},
  ]}
/>

The dashboard overlays two paths: the dashed vacuum model and the solid drag model. The important number is the range error. If the vacuum model says the shot clears the target but the drag model lands short, the problem is not "tuning." The problem is that the firmware was given the wrong world.

<Admonition type="warning" title="Hardware reality: coefficients are not universal">

The demo uses the coefficient values specified for this research case study. Do not copy them into a different FTC robot unless the projectile, wheel contact, launch height, and calibration procedure match. The method transfers; the constants have to be measured.

</Admonition>

## Step 3: The launcher is not a perfect energy converter

The firmware may start with a clean assumption:

$$
v_{\text{exit}} = \eta v_{\text{wheel}}
$$

where $\eta$ is the transfer efficiency between the flywheel surface and the projectile. In the research case study, the naive assumption was about $\eta = 0.70$, while the measured transfer was closer to $\eta \approx 0.26$ and changed with launch angle.

<EquationLegend
  items={[
    {symbol: '$v_{\\text{exit}}$', meaning: 'projectile speed as it leaves the launcher', unit: 'm/s'},
    {symbol: '$v_{\\text{wheel}}$', meaning: 'linear speed of the flywheel surface', unit: 'm/s'},
    {symbol: '$\\eta$', meaning: 'energy-transfer efficiency from wheel contact to projectile exit speed', unit: 'unitless'},
  ]}
/>

That gap is the difference between "the math says it should work" and "the robot actually scored." Contact time, compression, slip, asymmetric loading, and wheel scrub all live inside $\eta$. The simulator can expose the missing energy; calibration tells you how much is missing.

## Implementation sketch

The deployed version should keep physics and calibration separate. Physics predicts a first guess; calibration corrects the mechanism-specific loss.

```java
/**
 * Converts a desired projectile exit velocity into a flywheel command.
 * The transfer model is deliberately injected so it can be replaced after
 * calibration without rewriting the trajectory solver.
 */
public final class LauncherTransferModel {
    private final DoubleUnaryOperator efficiencyByAngleRadians;

    public LauncherTransferModel(DoubleUnaryOperator efficiencyByAngleRadians) {
        this.efficiencyByAngleRadians = efficiencyByAngleRadians;
    }

    public double wheelSurfaceSpeedFor(double exitVelocityMetersPerSecond, double launchAngleRadians) {
        double eta = efficiencyByAngleRadians.applyAsDouble(launchAngleRadians);
        if (eta <= 0.0) {
            throw new IllegalArgumentException("Transfer efficiency must be positive.");
        }
        return exitVelocityMetersPerSecond / eta;
    }
}
```
