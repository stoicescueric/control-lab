# On-Bot Control & Dynamic Targeting

<Difficulty level="Hard" />

<Abstract>
Once the projectile model exists, the robot still has to execute it under match conditions: a flywheel loses momentum after every shot, lookup tables can invent fake commands, and a moving chassis changes the aiming geometry. This lesson makes those three deployment problems interactive so the control logic can be inspected instead of trusted blindly.
</Abstract>

The simulator answers "what should the shot be?" The robot has a harder question: "can I make that shot right now?" If the flywheel is not recovered, if interpolation overshoots a calibration shelf, or if the robot aims at the real target while moving sideways, the paper model is technically correct and the shot still misses.

## Step 1: Taming the flywheel

A flywheel shot is a disturbance. The projectile leaves with energy that used to be stored in the wheel, so the measured velocity drops. A basic velocity controller tries to recover with error feedback:

$$
u = k_S + k_V r + k_P(r - y)
$$

    {symbol: 'k_V', meaning: 'velocity feedforward gain', unit: 'V per velocity'},

/>

The research controller adds an asymmetric recovery mode: if the wheel is below the ready band after a shot, command aggressively upward; if it is above target, do not kick downward with the same violence. That asymmetry matters because a flywheel below target causes a short shot, while a small overspeed can often be absorbed by the calibration margin.

## Step 2: Lookup tables can lie between points

Calibration data is sparse. The robot may know good shooter settings at a few distances, but during the match it needs values between those distances. A lookup table stores the measured shots and *interpolates* to fill the gaps — and the choice of interpolant decides whether those in-between values are safe.

<LookupTableIllustration />

A natural cubic spline is smooth, but smooth is not the same as safe. If the data contains a flat shelf, the spline can overshoot between equal points and command a speed that was never measured. The safer choice for monotone data is a monotone cubic Hermite interpolant, using the Fritsch-Carlson slope constraint:

$$
\alpha_k^2 + \beta_k^2 \le 9
$$

/>

The dashboard lets you drag calibration knots. The red curve shows what a natural spline can do in a dangerous flat region. The blue curve shows the monotone Hermite alternative staying inside the behavior implied by the data.

## Step 3: Shoot-on-the-move is a fixed-point problem

When the robot drives while firing, the projectile inherits chassis velocity. The turret should not aim at the real target; it should aim at a virtual target shifted by the robot motion during the projectile's flight:

$$
\mathbf{p}^{(i+1)}_{\text{virtual}}
=
\mathbf{p}_{\text{target}}
+
G\mathbf{v}_{R}\left(t_d + t_f^{(i)}\right)
$$

<EquationLegend
  items={[
    {symbol: 'p_virtual', meaning: 'virtual point the turret aims at', unit: 'field position'},
    {symbol: 'p_target', meaning: 'fixed real target location', unit: 'field position'},
    {symbol: 'G', meaning: 'empirical lead gain from the research controller', unit: 'unitless'},
    {symbol: 'v_R', meaning: 'robot chassis velocity vector', unit: 'distance/s'},
    {symbol: 't_d', meaning: 'feeder or launch delay before free flight', unit: 's'},
    {symbol: 't_f', meaning: 'projectile flight time from lookup or simulator', unit: 's'},
  ]}
/>

This is circular: the virtual target changes the distance, the distance changes flight time, and flight time changes the virtual target. Fixed-point iteration breaks the loop. Start with the real target, compute a flight time, shift the target, compute the new flight time, and stop when the time changes by less than the tolerance.

It converges fast for a concrete reason: the lead distance $G v_R (t_d + t_f)$ is small next to the shot distance, and a small change in distance produces an even smaller change in flight time, so each pass moves the answer less than the last. The map is a contraction, and in the dashboard's solver trace you can watch $\Delta t_f$ shrink below the tolerance in two or three rounds. The iteration cap is there only to bound the real-time loop if a bad lookup table breaks that contraction — not because convergence is in doubt under normal conditions.

## How the deployed solver is structured

On the robot the fixed-point loop is made explicit and *bounded*. It runs inside the autonomous control loop, so it is given a hard iteration cap: even if a bad table or impossible geometry breaks convergence, the loop must still return in predictable time rather than spin.

```text
solve_virtual_target(shooter, target, robot_velocity):
    flight_time <- lookup_flight_time( distance(shooter, target) )

    repeat at most MAX_ITERATIONS times:
        lead_time <- feeder_delay + flight_time
        virtual   <- target + lead_gain * robot_velocity * lead_time

        next_time <- lookup_flight_time( distance(shooter, virtual) )
        if | next_time - flight_time | < tolerance:
            stop        # flight time stopped moving: converged
        flight_time <- next_time

    return virtual
```

Two parameters carry the whole behaviour: the convergence `tolerance` decides when "close enough" is reached, and `MAX_ITERATIONS` guarantees the loop terminates regardless. The flight-time lookup is the calibrated table from Step 2, which is exactly why it must be monotone — a table that overshoots between points could break the contraction this loop relies on.

<Admonition type="warning" title="Hardware reality: ready is a band, not one number">

Do not wait for an encoder reading to equal the target exactly. A match robot should use a readiness band, because encoder quantization, loop timing, shot disturbance, and battery sag make exact equality a trap.

</Admonition>
