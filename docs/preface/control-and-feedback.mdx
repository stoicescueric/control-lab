# Control & Feedback

<Difficulty level="Easy" />

<Abstract>
Control engineering is the discipline of making a physical system reach and hold a desired behavior while the real world pushes back.
Feedback is the central idea: measure the system, compare it to the target, and use that error to compute the next command.
</Abstract>

Control is not a PID acronym. PID is only one possible rule inside a much larger pattern.

Every robot mechanism has the same basic story:

1. You want something to happen.
2. The mechanism does something close, but not perfect.
3. A sensor tells you what actually happened.
4. Code decides the next motor command.

That loop is the foundation behind arm control, flywheel velocity, drivetrain heading hold, odometry
correction, path following, and state estimation.

## FTC example: an arm does not stop where power says it should

Imagine an arm that scores a game element. During testing, you discover that `0.28` motor power holds
the arm near the target angle when the battery is full and the robot is sitting still.

Then the match starts.

The battery voltage drops. The intake has a game element in it. A cable rubs slightly against the
mechanism. The same `0.28` power no longer holds the same angle. The robot did not become mysterious;
the code was open-loop. It sent a command without checking whether the command worked.

Feedback fixes that failure by turning the mechanism into a conversation:

> "Where should you be? Where are you now? How hard should I push next?"

## Open loop is not wrong; it is just blind

Open-loop control means you send an output without using the result to change the next output. A
time-based autonomous drive is open loop:

<JavaCode league="FTC" label="Open-loop drivetrain timing">{`frontLeft.setPower(0.5);
frontRight.setPower(0.5);
backLeft.setPower(0.5);
backRight.setPower(0.5);

sleep(700);

frontLeft.setPower(0.0);
frontRight.setPower(0.0);
backLeft.setPower(0.0);
backRight.setPower(0.0);`}</JavaCode>

An intake that simply runs while the driver holds a trigger is also open loop:

<JavaCode league="FTC" label="Open-loop intake">{`if (gamepad1.right_trigger > 0.2) {
    intakeMotor.setPower(1.0);
} else {
    intakeMotor.setPower(0.0);
}`}</JavaCode>

This is often fine. An intake may not need to land at a precise state. A simple timed motion may be
good enough for a low-risk autonomous action. The question is not "is open loop bad?" The question is:
what are the design requirements?

Ask:

- Does the mechanism need accuracy, or only motion?
- Will battery voltage, load, or defense contact change the result?
- Is there a sensor that can measure success?
- Is a bad final position costly during a match?

If the answer is "yes" to accuracy, disturbances, measurable success, or high cost, you probably want
feedback.

## Theoretical rigor

A feedback system starts with a target and a measurement.

$$
e = r - y
$$

Here, $e$ is error, $r$ is the reference or setpoint, and $y$ is the measured output. For an FTC arm,
$r$ might be the desired arm angle and $y$ might be the encoder-estimated arm angle.

The controller maps error into a command:

$$
u = C(e)
$$

Here, $u$ is the actuator command and $C(\cdot)$ is the controller rule. That rule could be as simple
as proportional control or as advanced as a model-based controller.

The robot mechanism then responds:

$$
y_{k+1} = P(y_k, u_k, d_k)
$$

Here, $k$ is the current loop number, $k+1$ is the next loop, $P(\cdot)$ represents the physical plant
or mechanism, and $d_k$ is a disturbance such as battery sag, friction, defense contact, gravity, or a
game element entering the mechanism.

The key difference is whether the next command depends on the latest measurement.

| Style | Command | What it can handle |
|---|---|---|
| Open loop | $u_k = u_\text{fixed}$ | only the condition it was tuned for |
| Closed loop | $u_k = C(r_k - y_k)$ | changes in load, voltage, friction, and position |

Open loop can still be useful. Feedforward is often an open-loop prediction from a model. But by
itself, open loop cannot know when it is wrong.

Closed loop is useful because it makes failure observable. If the measured arm angle is low, the error
is positive. If the flywheel slows after shooting, the velocity error grows. If the drivetrain heading
drifts, the heading error appears. The controller does not need to guess that reality changed; the
sensor makes the change visible.

### Two first controllers

The simplest closed-loop controller is a bang-bang controller. It looks at the error and chooses one of
two outputs:

$$
u =
\begin{cases}
u_\text{up}, & e > \epsilon \\
u_\text{down}, & e < -\epsilon \\
0, & |e| \le \epsilon
\end{cases}
$$

Here, $\epsilon$ is a tolerance band. Without a tolerance, a real robot may wait forever for the
measurement to equal the setpoint exactly, which almost never happens.

<JavaCode league="FTC" label="Bang-bang arm hold with tolerance">{`double error = targetTicks - armMotor.getCurrentPosition();
double toleranceTicks = 20.0;

if (error > toleranceTicks) {
    armMotor.setPower(0.45);
} else if (error < -toleranceTicks) {
    armMotor.setPower(-0.25);
} else {
    armMotor.setPower(0.0);
}`}</JavaCode>

Bang-bang is easy to understand, but it often oscillates because the command changes abruptly. A
proportional controller is the next step:

$$
u = k_P e
$$

Here, $k_P$ is the proportional gain. A larger error creates a larger command; a smaller error creates
a smaller command.

<JavaCode league="FTC" label="Proportional arm controller">{`double error = targetTicks - armMotor.getCurrentPosition();
double command = kP * error;

armMotor.setPower(Range.clip(command, -1.0, 1.0));`}</JavaCode>

This proportional rule is the first piece of PID. Later, the PID lesson adds memory of past error and
reaction to changing error, but the core feedback idea is already here.

## Enterprise implementation

Production robot code should make the feedback loop explicit. Sensors read. Controllers compute.
Actuators write. Telemetry records what happened.

<JavaCode league="FTC" label="A small feedback-control architecture">{`/**
 * Reads the current value controlled by a feedback loop.
 */
public interface FeedbackSensor {
    /**
     * @return measured process value in real engineering units.
     */
    double getMeasurement();
}

/**
 * Applies a normalized command to hardware.
 */
public interface Actuator {
    /**
     * @param command normalized actuator command in the range [-1.0, 1.0].
     */
    void setCommand(double command);
}

/**
 * Converts setpoint and measurement into an actuator command.
 */
public interface FeedbackController {
    /**
     * @param setpoint desired value in the same units as measurement.
     * @param measurement current measured value.
     * @param dtSeconds elapsed loop time in seconds.
     * @return normalized actuator command before final safety clamping.
     */
    double calculate(double setpoint, double measurement, double dtSeconds);
}

public final class FeedbackLoop {
    private final FeedbackSensor sensor;
    private final Actuator actuator;
    private final FeedbackController controller;
    private double setpoint;

    public FeedbackLoop(
            FeedbackSensor sensor,
            Actuator actuator,
            FeedbackController controller) {
        this.sensor = sensor;
        this.actuator = actuator;
        this.controller = controller;
    }

    public void setSetpoint(double setpoint) {
        this.setpoint = setpoint;
    }

    /**
     * Runs one control iteration. Call this exactly once per OpMode loop.
     */
    public void update(double dtSeconds, Telemetry telemetry) {
        double measurement = sensor.getMeasurement();
        double command = controller.calculate(setpoint, measurement, dtSeconds);
        double safeCommand = Range.clip(command, -1.0, 1.0);

        actuator.setCommand(safeCommand);

        telemetry.addData("setpoint", setpoint);
        telemetry.addData("measurement", measurement);
        telemetry.addData("error", setpoint - measurement);
        telemetry.addData("command", safeCommand);
    }
}`}</JavaCode>

This style makes later lessons easier to plug in. A proportional controller, PID controller,
feedforward-plus-PID controller, or motion-profiled controller can all implement the same
`FeedbackController` interface.

## Practice

Try these on a safe test stand or with the robot wheels off the ground first:

- Run a drivetrain forward for a fixed time. Does the distance change between a full battery and a
  low battery?
- Move an arm with bang-bang control and a tolerance. Does it chatter near the setpoint?
- Replace bang-bang with proportional control. What value of $k_P$ starts to oscillate?
- Add telemetry for setpoint, measurement, error, and command. Which value tells the clearest story
  when behavior is wrong?

## Hardware reality

<Admonition type="warning" title="Feedback is only as good as the measurement">

If the encoder slips, the IMU drifts, or the camera frame arrives late, the controller is reacting to
bad information. Feedback does not magically make a robot correct; it makes the robot correctable when
the measurement is trustworthy enough.

</Admonition>

<Admonition type="warning" title="Closed loop does not remove actuator limits">

If the motor is already at full power, asking for more command cannot create more voltage. Saturation
changes controller behavior, especially for integral terms, so real implementations need output
clamps, anti-windup, current limits, and carefully chosen motion targets.

</Admonition>

<Admonition type="tip" title="FTC intuition">

When a mechanism fails only after the battery drops, load changes, or a defender bumps the robot, ask
whether the code can measure the failure. If it cannot measure the failure, it cannot correct it.

</Admonition>
