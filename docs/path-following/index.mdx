# Module 5 · Path Following & Kinematics

Autonomous movement is where localization and control finally pay off. A great autonomous routine does
two things well: it can **drive to any pose on the field**, and it can **follow a smooth path** through
many poses at speed without stopping to re-aim. Everything in this module builds toward those two
capabilities.

<Admonition type="note" title="Prerequisites">

This module *composes* the two before it, so have them in hand first:

- **[Localization & Odometry](../localization-odometry/odometry.mdx)** — every follower here assumes a
  trustworthy pose $(x, y, \theta)$ updated each loop.
- **[Control Theory](../control-theory/index.mdx)** — the [PID](../control-theory/pid.mdx) and
  [feedforward](../control-theory/feedforward.mdx) loops that turn an error into a motor command.

If a follower misbehaves, the bug is usually in the *pose estimate* or the *underlying controller*, not
the path logic on top.

</Admonition>

## The autonomous-movement pipeline

Every method in this module — from the simplest go-to-pose to a guided vector field — is the same five
steps run every loop. The methods differ only in *how* they do steps 2 and 3.

| Step | What happens | Where it lives |
|---|---|---|
| 1. **Localize** | read the current pose from odometry / vision | [Module 3](../localization-odometry/odometry.mdx) |
| 2. **Plan the target** | pick where to go next — a target pose, a lookahead point, or a field vector | this module |
| 3. **Control** | turn the error into a chassis command $(v_x, v_y, \omega)$ with PID + feedforward | this module + [Module 4](../control-theory/index.mdx) |
| 4. **Mix & correct** | convert the chassis command to wheel powers, desaturate, re-run next loop | [Mecanum kinematics](./mecanum-kinematics.mdx) |
| 5. **Terminate** | stop when within a pose tolerance *and* slow — or when a timeout fires | every follower |

<Admonition type="tip" title="Step 5 is not optional">

A follower that never decides it is "done" will hunt around the target forever, and one that trusts only
position will report done while still coasting. Always gate completion on **pose tolerance AND low
velocity**, with a **timeout** as a backstop so a stuck routine yields the field.

</Admonition>

## From simple to advanced

The honest way to learn this is in order of capability. Start with the follower you can debug by eye, and
only add machinery when a real limitation forces you to.

1. **[Mecanum kinematics](./mecanum-kinematics.mdx)** — the matrix that turns one chassis command into
   four wheel powers, plus field-centric driving and joystick response curves. *This is step 4 of the
   pipeline; build it first so every later method has a drivetrain to command.*
2. **[PID-to-point](./pid-to-point.mdx)** — the simplest complete follower: three PID loops drive the
   robot to a single target pose. Most autonomous routines are just a list of these.
3. **[Angle wrapping](./angle-wrapping.mdx)** — the small, mandatory fix that stops the robot spinning
   the long way around.
4. **[Predictive deceleration](./predictive-deceleration.mdx)** — cap speed by the distance left to stop,
   so the robot eases onto the point instead of overshooting it.
5. **[Quintic & Bézier splines](./quintic-splines.mdx)** — the two ways to describe a smooth, curvature-
   continuous path: Bézier control points (approximation) versus quintic Hermite waypoints
   (interpolation).
6. **[Pure pursuit](./pure-pursuit.mdx)** — chase a lookahead "carrot" along a path; fixed versus dynamic
   lookahead.
7. **[Guided vector fields](./guided-vector-fields.mdx)** — follow a path as a flow that converges onto
   the path, rather than chasing a moving point.

Steps 1–2 give you a working autonomous today; 3–4 make it reliable; 5–7 make it fast and smooth. Read
them in order — kinematics gives you a drivetrain you can command, the next three turn one target into
clean motion, and the last three follow whole paths at speed.
