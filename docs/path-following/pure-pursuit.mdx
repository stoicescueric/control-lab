# Path Following (Pure Pursuit)

<Difficulty level="Hard" />

<Abstract>
Pure pursuit turns a planned path into smooth steering with one geometric rule: continuously aim at a
"carrot" a fixed look-ahead distance along the path. This lesson derives the curvature that arc
requires and shows how the look-ahead distance trades tracking tightness against smoothness.
</Abstract>

The robot knows where it is (odometry) and where it should go (a planned path). Path following is
the controller that connects them: it converts the path into a continuous stream of drive commands.
**Pure Pursuit** is the classic, deployable method — a single geometric rule that has steered FTC
and FRC robots for years.

<Problem title="Waypoint-to-waypoint driving is slow and discontinuous">

The naïve approach drives straight to waypoint 1, stops, rotates, drives to waypoint 2, and so on.
It is slow, jerky, and stresses the drivetrain at every stop. Competition autonomous needs to follow
a *continuous* curve at speed, correcting smoothly for the inevitable pose error rather than halting
to re-aim. The controller must answer, every loop: given my current pose and this path, what
curvature should I drive?

<Analogy>
<strong>A carrot on a stick.</strong> Fix a carrot a constant distance ahead of you, always on the
path, and simply walk toward it. As you advance, the carrot slides further along the path, so you
continuously curve to follow it. That moving target is the <strong>lookahead point</strong>.
</Analogy>

</Problem>

<Theory title="One arc to a lookahead point">

The algorithm is three steps, run every loop:

1. **Find the lookahead point** — walk along the path until you are a fixed lookahead distance $L_d$
   away. That point is the carrot.
2. **Transform it into the robot frame**, where $+x$ is forward and $+y$ is left. Call the lateral
   offset $y$.
3. **Compute the arc** — exactly one circular arc starts at the robot (tangent to its heading) and
   passes through the carrot. Its curvature, and the resulting turn rate at speed $v$, are:

$$
\kappa = \frac{2y}{L_d^{\,2}}, \qquad \omega = \kappa\,v
$$

<Formula>
  curvature = <Rose>2·y / Ld²</Rose> &nbsp;&nbsp; turn rate <Amber>ω = curvature · v</Amber>
</Formula>

This drops out of geometry: the chord to the carrot has length $L_d$ and lateral offset $y$, so the
unique tangent circle through both has radius $r = L_d^2 / (2y)$, hence curvature $1/r = 2y/L_d^2$.
If the carrot is dead ahead ($y = 0$), curvature is zero — drive straight.

The single tuning parameter is the lookahead distance, and it is a direct trade-off:

| Lookahead $L_d$ | Behaviour |
|---|---|
| **Too small** | tracks tightly but oscillates — over-corrects and weaves |
| **Too large** | very smooth but cuts corners and rounds off tight features |
| **Tuned** | smooth tracking with stable, gentle correction |

In the demo, the robot follows the blue path; the <span style={{ color: "#ffc24d" }}>amber
carrot</span> is its lookahead target. Drag the waypoints to reshape the path and sweep the lookahead
from small (watch it oscillate) to large (watch it cut corners).

the geometry the path demands.
</Callout>

### Finding the carrot precisely: line–circle intersection

"Walk along the path until you are $L_d$ away" is fine for densely-sampled waypoints, but the exact
carrot is where a **circle of radius $L_d$ around the robot crosses the path**. On a straight segment
from $\mathbf{A}$ to $\mathbf{B}$, parametrize the point as $\mathbf{E}(t) = \mathbf{A} + t(\mathbf{B} -
\mathbf{A})$ and solve $\lVert \mathbf{E}(t) - \mathbf{P} \rVert^2 = L_d^2$ for the robot at $\mathbf{P}$.
With $\mathbf{d} = \mathbf{B} - \mathbf{A}$ and $\mathbf{f} = \mathbf{A} - \mathbf{P}$ that is a quadratic
in $t$:

$$
(\mathbf{d}\cdot\mathbf{d})\,t^2 + 2(\mathbf{f}\cdot\mathbf{d})\,t + (\mathbf{f}\cdot\mathbf{f} - L_d^2) = 0
$$

Solve with the quadratic formula, keep the roots in $[0, 1]$, and take the **larger** one — the
intersection furthest along the segment, so the carrot always moves forward. If the discriminant is
negative the circle does not reach the path (robot too far off): fall back to the closest path point so
the robot drives back toward the line.

<JavaCode league="FTC" label="Lookahead point as a segment–circle intersection">{`/** Intersection of the lookahead circle (center P, radius Ld) with segment A->B.
 *  Returns the furthest-forward hit in [0,1], or null if the circle misses. */
static Double intersectT(Vector2d A, Vector2d B, Vector2d P, double Ld) {
    Vector2d d = B.minus(A);
    Vector2d f = A.minus(P);
    double a = d.dot(d);
    double b = 2 * f.dot(d);
    double c = f.dot(f) - Ld * Ld;
    double disc = b * b - 4 * a * c;
    if (disc < 0) return null;                 // circle doesn't reach the segment
    disc = Math.sqrt(disc);
    double t2 = (-b + disc) / (2 * a);          // furthest-forward root first
    double t1 = (-b - disc) / (2 * a);
    if (t2 >= 0 && t2 <= 1) return t2;
    if (t1 >= 0 && t1 <= 1) return t1;
    return null;
}`}</JavaCode>

### Fixed vs. dynamic lookahead

The table above is the case *against* a single fixed lookahead: the value that smooths a fast straight
is the same value that cuts a tight corner. A **fixed** lookahead picks one compromise for the whole
path. A **dynamic** lookahead changes $L_d$ as the robot drives, escaping the trade-off:

- **Speed-scaled** — make lookahead grow with speed, $L_d = \operatorname{clamp}(k\,v,\ L_\text{min},\ L_\text{max})$. Long and smooth on fast straights, short and tight as the robot slows for a corner. This mirrors how a driver looks farther down the road the faster they go.
- **Curvature-scaled** — shrink lookahead where the upcoming path bends sharply (large $|\kappa|$ from the [spline](./quintic-splines.mdx)) and grow it on gentle sections, so tight features are tracked and straights stay calm.

$$
L_d = \operatorname{clamp}\big(k_v\, v,\ L_\text{min},\ L_\text{max}\big)
$$

Dynamic lookahead is strictly more capable but adds knobs to tune, and a lookahead that changes too
abruptly can itself induce steering jitter — rate-limit it. If you find yourself fighting a fixed
lookahead on a path with both fast and tight sections, that is the signal to make it dynamic, or to
switch to a [guided vector field](./guided-vector-fields.mdx) that drops the lookahead idea entirely.

### Adaptive pure pursuit: plan the speed, not just the steering

Plain pure pursuit answers *where to steer*. **Adaptive** pure pursuit — the version Team 1712 (DAWGMA)
popularised in FRC — adds the other half: *how fast to go at every point*, decided ahead of time from
the path's own geometry. The robot then "adapts" its speed to the curve, braking for tight turns and
opening up on straights, with no hand-authored velocity plan. It is a three-pass precompute over the
densely-sampled path, plus a one-line runtime limit.

**1 — Curvature at every point.** Take each point with its two neighbours $P_{i-1}, P_i, P_{i+1}$ and
fit the unique circle through them; the curvature is $\kappa_i = 1/R$. The compact way to get $R$ is the
circumscribed-circle (Menger) formula — with triangle side lengths $a, b, c$ and area $A$:

$$
R = \frac{a\,b\,c}{4A}, \qquad \kappa_i = \frac{1}{R} = \frac{4A}{a\,b\,c},
\qquad A = \tfrac12\big\lvert (P_i - P_{i-1}) \times (P_{i+1} - P_i)\big\rvert
$$

A straight run gives $A = 0$, hence $\kappa = 0$ (infinite radius); the two endpoints have no neighbour
pair, so their curvature is set to zero.

Drag the three points below to feel it: the tighter you bend them, the smaller the circle and the larger
$\kappa$ — pull them into a line and the curvature collapses to zero.

<CurvatureCircle />

**2 — A speed cap from curvature.** Slow down where the path bends. Each point's ceiling is the path's
max speed, lowered wherever curvature is high:

$$
v_i^{\max} = \min\!\left(v_\text{path},\ \frac{k_\kappa}{\kappa_i}\right)
$$

The constant $k_\kappa$ (typically $1$–$5$) sets how cautious the robot is in turns; as $\kappa \to 0$
the cap relaxes back to $v_\text{path}$. This is the discrete cousin of the
[curvature speed limit](./quintic-splines.mdx) $v \le \sqrt{a_\text{lat} R}$ from the splines lesson.

**3 — A backward pass so it can always stop.** A speed cap alone lets the robot reach a slow corner
still going full tilt. Walk the path *backwards* from the end (where $v = 0$) and clamp each point to the
fastest speed from which it could still decelerate to its successor — the very
[braking kinematics](./pid-to-point.mdx) $v_f = \sqrt{v_i^2 + 2 a d}$, run over the path
offline:

$$
v_i \leftarrow \min\!\left(v_i,\ \sqrt{v_{i+1}^2 + 2\,a_\text{max}\,d_i}\right)
$$

with $d_i$ the distance between consecutive points. After this pass every stored velocity is guaranteed
decelerable to a stop at the end.

**4 — A forward rate limit at runtime.** The backward pass deliberately leaves *acceleration* unbounded
— a forward pass would force the start point to $0$ and the robot would never move. Instead, cap how fast
the commanded speed may change each loop:

$$
\lvert \Delta v\rvert \le a_\text{max}\,\Delta t
$$

a one-line [rate limiter](../signal-processing/low-pass.mdx) applied to the target velocity the robot
pulls from its closest point. The backward pass plans the braking; the rate limiter shapes the
acceleration.

Here is the whole plan, live. Drag the waypoints into a tight hairpin and watch the path turn red there
as the **curvature cap** (blue, right) pulls the speed down; the **backward pass** (orange) then bends
the profile down to zero at the end so the robot can always stop. The sliders are the three knobs from
the equations above.

### Building the path: inject, then smooth

Adaptive pure pursuit needs a *dense, smooth* list of points — the curvature and velocity passes are
only as good as the spacing. You can get one from the [splines](./quintic-splines.mdx) in the previous
lesson, or with the lighter trick the DAWGMA paper uses:

1. **Inject** evenly-spaced points along each straight segment between your hand-placed waypoints (a
   spacing of ~6 in works well), turning a few corners into hundreds of closely-spaced points.
2. **Smooth** them with a gradient-descent pass that trades staying near the original points against
   straightening local bends, iterated to convergence:

$$
P_i \leftarrow P_i + \underbrace{a\,(P_i^\text{orig} - P_i)}_{\text{stay near input}}
  + \underbrace{b\,(P_{i-1} + P_{i+1} - 2P_i)}_{\text{pull toward neighbours}}
$$

with $b$ near $0.75$–$0.98$ (larger = smoother), $a = 1 - b$, repeated until the total change drops below
a tolerance. It is a cheap alternative to fitting a spline — no overshoot, though it does pull slightly
away from your original waypoints.

</Theory>

<Deploy title="The loop, and what production followers add">

One iteration on the Control Hub: locate the carrot ahead, transform it into the robot frame, convert
to curvature, and command forward speed plus turn rate.

<JavaCode league="FTC" label="Pure pursuit, one loop iteration">{`Pose2d robot = drive.localizer.getPose();

// 1. Lookahead: first path point at least Ld ahead (never target behind us).
Vector2d carrot = path.get(path.size() - 1);
for (int i = lastIndex; i < path.size(); i++) {
    if (robot.position.minus(path.get(i)).norm() >= LOOKAHEAD) {
        carrot = path.get(i);
        lastIndex = i;
        break;
    }
}

// 2. Carrot in the robot frame (+x forward, +y left).
Vector2d d = carrot.minus(robot.position);
double h = robot.heading.toDouble();
double yLocal = -Math.sin(h) * d.x + Math.cos(h) * d.y;
double L = d.norm();

// 3. Arc curvature -> drive command.
double curvature = 2 * yLocal / (L * L);
drive.setDrivePowers(CRUISE_SPEED, curvature * CRUISE_SPEED);`}</JavaCode>

Pure Pursuit is the foundation. Road Runner and Pedro Pathing follow full <em>trajectories</em> —

For the **adaptive** velocity plan, precompute it once over the dense path: curvature per point, a
curvature speed cap, then the backward braking pass.

<JavaCode league="FTC" label="Adaptive velocity profile (offline, over the dense path)">{`/** Fill each point's curvature, then cap and back-propagate its speed. */
void buildVelocityProfile(List<PathPoint> pts, double vPath, double kCurv, double aMax) {
    // 1) curvature from the circle through each triple (endpoints stay 0)
    for (int i = 1; i < pts.size() - 1; i++) {
        pts.get(i).curvature = mengerCurvature(pts.get(i - 1).pos, pts.get(i).pos, pts.get(i + 1).pos);
    }
    // 2) speed cap: slower where the path bends
    for (PathPoint p : pts) {
        p.velocity = (p.curvature < 1e-6) ? vPath : Math.min(vPath, kCurv / p.curvature);
    }
    // 3) backward pass: guarantee every point can still brake to the end
    pts.get(pts.size() - 1).velocity = 0;
    for (int i = pts.size() - 2; i >= 0; i--) {
        double d = pts.get(i).pos.minus(pts.get(i + 1).pos).norm();
        double reachable = Math.sqrt(pts.get(i + 1).velocity * pts.get(i + 1).velocity + 2 * aMax * d);
        pts.get(i).velocity = Math.min(pts.get(i).velocity, reachable);
    }
}

/** 4*Area / (a*b*c): curvature of the circle through three points (0 if collinear). */
static double mengerCurvature(Vector2d A, Vector2d B, Vector2d C) {
    double area = 0.5 * Math.abs((B.x - A.x) * (C.y - A.y) - (C.x - A.x) * (B.y - A.y));
    double a = B.minus(C).norm(), b = A.minus(C).norm(), c = A.minus(B).norm();
    double denom = a * b * c;
    return denom < 1e-9 ? 0 : 4 * area / denom;
}

// runtime: pull the closest point's planned speed, then rate-limit how fast it may change
double target  = pts.get(closestIndex).velocity;
double maxStep = A_MAX * loopDt;
commanded += Math.max(-maxStep, Math.min(maxStep, target - commanded));`}</JavaCode>

<Callout type="idea" title="On a tank (differential) drive">
Pure pursuit outputs a signed curvature and a speed; a differential drive turns those straight into
wheel speeds. With track width $T$, $\;L = v(2 + \kappa T)/2\;$ and $\;R = v(2 - \kappa T)/2$. (Our
mecanum stack instead feeds the same $(v,\ \omega = \kappa v)$ into the
[mixer](./mecanum-kinematics.mdx).) Use a track width a few inches larger than the real one to absorb
turning scrub.
</Callout>

I do" half. The curriculum culminates in a research-grade application of all of it.
