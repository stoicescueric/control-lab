# PID Control

<Difficulty level="Medium" />

<Abstract>
PID is the workhorse feedback controller: it sums three reactions to the error — proportional to its
present value, its accumulated past, and its rate of change — into a single command. This lesson builds
each term from intuition — including *why* the derivative term lands you on the target with zero velocity —
then upgrades the loop to the FTC "PDFL" controller: a constant feedforward and a static-friction floor
that replace the integral entirely, plus a low-pass-filtered derivative.
</Abstract>

PID is the controller you will deploy most often: arm position, flywheel velocity, drivetrain
heading, lift height. It is three reactions to a single error signal, summed into one command. The
name is intimidating; the mechanism is not.

<Problem title="Proportional-only control leaves a standing error">

Consider an arm that must hold 90° against gravity. The intuitive controller is *proportional* —
command in proportion to the error, $u = K_p e$. It has a structural flaw: to produce the steady
holding torque gravity demands, it **requires a non-zero error to exist**. The arm settles a few
degrees low and stays there. Raising $K_p$ shrinks the gap but invites overshoot and oscillation,
and it still never reaches zero. Disturbances — a game element's weight, a sagging battery — make it
worse. We need terms that address the error's *history* and its *trend*, not just its present value.

</Problem>

<Theory title="Three terms acting on present, past, and future">

The error is the gap between target and measurement, $e(t) = \text{setpoint} - \text{measurement}$.
PID sums three responses to it:

- **P — proportional** · *the present.* $u_P = K_p\,e(t)$ — push in proportion to the current gap.
- **I — integral** · *the past.* $u_I = K_i \int_0^t e(\tau)\,d\tau$ — accumulate effort until a
  persistent gap is eliminated. This is what removes the standing error above.
- **D — derivative** · *the future.* $u_D = K_d \frac{de(t)}{dt}$ — anticipate: if the gap is
  closing quickly, ease off early to avoid overshoot.

$$
u(t) = K_p\,e(t) + K_i \int_0^t e(\tau)\,d\tau + K_d \frac{de(t)}{dt}
$$

<Callout type="idea" title="What the three terms really are: a mass on a spring.">
<ul>
<li><strong>P is the spring.</strong> It pulls toward the target with a force proportional to how far off
you are. A spring on its own overshoots and oscillates forever.</li>
<li><strong>D is the shock absorber.</strong> It opposes <em>motion</em>. Because the well-behaved version
differentiates the measurement, $u_D = -K_d\,\dot m$ is literally <strong>velocity feedback</strong>: the
faster you're moving, the harder it brakes. So D does more than "reduce overshoot" — it makes the system
<strong>arrive at the target with ≈ zero velocity</strong> instead of barrelling through it. P + D
together is a critically-damped mass-spring-damper: a clean settle with no overshoot.</li>
<li><strong>I shifts the rest position.</strong> It slowly builds whatever constant push the job needs
(holding an arm against gravity) so the spring no longer has to hold a standing error to make that
force.</li>
</ul>
</Callout>

The gains $K_p, K_i, K_d$ are the tunable parameters. The drone below must hover at the dashed
target while gravity pulls it down — the canonical "standing error" problem. Drag the target, tune
the gains, and read the live contribution of each term.

<Drone />

<Callout type="idea" title="Three experiments that isolate each term:">
<ol>
<li><strong>"P only"</strong> — the drone stabilises <em>below</em> target. With no integral term it
cannot generate the constant thrust that cancels gravity without holding an error. This is
steady-state error, made visible.</li>
<li><strong>"PID (tuned)"</strong> — the integral term accumulates exactly enough extra thrust to
cancel gravity; the drone settles on target. Watch the yellow I-meter climb.</li>
<li><strong>"Too much Kp"</strong> + a <strong>wind gust</strong> — overshoot and oscillation. The
derivative term is what damps it.</li>
</ol>
</Callout>

**Tuning** is the engineering. A reliable procedure: raise $K_p$ until the mechanism just begins to
oscillate, then back off; add $K_d$ to damp the overshoot; add $K_i$ last, only if a standing error
remains.

| Symptom | Likely cause | Correction |
|---|---|---|
| Slow to respond | `kP` too low | raise `kP` |
| Overshoots & oscillates | `kP` high / `kD` low | lower `kP`, raise `kD` |
| Never reaches target | no / low `kI` | add a little `kI` |
| Slow drift, sluggish wobble | `kI` too high (windup) | lower `kI` |
| Jittery, twitchy | `kD` too high (noise) | lower `kD`, filter the input |

</Theory>

<Deploy title="A discrete PID loop, hardened for the Control Hub">

The hub can't integrate or differentiate, so discretise: the integral becomes a running sum, the
derivative a difference over the measured $\Delta t$.

$$
u_k = K_p e_k + K_i \sum_{j} e_j \Delta t + K_d \frac{e_k - e_{k-1}}{\Delta t}
$$

Two refinements are mandatory on real hardware:

- **Derivative on measurement.** Differentiating the *error* spikes the output whenever the setpoint
  changes ("derivative kick"). Differentiate the *measurement* instead — $-K_d\,\dot{m}$ — for the
  same damping without the kick. (The drone above already does this with $-K_d\,\dot{y}$.)
- **Anti-windup.** When the actuator is saturated, stop accumulating the integral; otherwise it
  "winds up" and causes a large overshoot once the target is reached.

FTCLib provides the controller, and FTC Dashboard tunes the gains live over Wi-Fi:

<JavaCode league="FTC" label="FTCLib + Dashboard, with the refinements built in">{`@Config            // exposes the gains to FTC Dashboard for live tuning
public class ArmController {
    public static double kP = 0.012, kI = 0.0, kD = 0.0006;

    private final DcMotorEx motor;
    private final PIDController pid = new PIDController(kP, kI, kD);

    public ArmController(HardwareMap hw) {
        motor = hw.get(DcMotorEx.class, "arm");
        motor.setMode(DcMotor.RunMode.RUN_WITHOUT_ENCODER);
    }

    public void update(int targetTicks) {
        pid.setPID(kP, kI, kD);                       // pick up Dashboard edits
        double power = pid.calculate(motor.getCurrentPosition(), targetTicks);
        motor.setPower(Math.max(-1, Math.min(1, power)));   // saturate safely
    }
}`}</JavaCode>

<Callout type="tip" title="Filters and controllers are a team.">
The derivative term amplifies sensor noise. Production controllers run the measurement through a
low-pass filter first — a noisy encoder or distance sensor will wreck a D term. Clean the signal,
then control it.
</Callout>

### Beyond textbook PID: the PDFL controller

Notice the arm problem never actually needed an *integral*. The steady push gravity demands is
*predictable*, so you can supply it directly and skip the slow, windup-prone I term. That is the idea
behind the **PDFL** controller used widely in FTC — **P**roportional, **D**erivative, **F**eedforward, and
an **L**ower-limit (static-friction) term, with **no integral at all**:

- **P** — proportional to the error, exactly as before.
- **D** — the derivative, **low-pass filtered** (an exponential moving average) so encoder noise doesn't
  wreck it; the filter lives *inside* the D term (see [low-pass filters](../signal-processing/low-pass.mdx)).
  It is still the velocity feedback that brings you in at zero velocity.
- **F** — a **constant** feedforward: the steady power that holds the mechanism in place against gravity.
  It does the integral's old job instantly, with nothing to wind up.
- **L** — a **static-friction floor**: a fixed $k_L\,\operatorname{sign}(e)$ that shoves *just hard enough*
  to break stiction in the direction you need to move. It is applied only **outside a small dead zone**
  around the target — inside it the sign of the error flips every loop, so dropping L there stops the
  mechanism from buzzing.

$$
u = \underbrace{k_P\,e}_{P} + \underbrace{k_D\,\dot e_\text{filtered}}_{D}
  + \underbrace{k_F}_{F} + \underbrace{k_L\,\operatorname{sign}(e)}_{L\ \text{(outside the dead zone)}}
$$

<Callout type="idea" title="How to actually find F and L.">
$k_F$ is the term you'd like to tune first, but it is hard to read off directly. The trick: ease the
command up by hand until you find the power <strong>just as the mechanism starts to move up</strong>
($u_\text{up}$), then the power <strong>just as it starts to move down</strong> ($u_\text{down}$). Going
up you fight gravity <em>and</em> friction ($u_\text{up} = k_F + k_L$); going down gravity helps while
friction still resists ($u_\text{down} = k_F - k_L$). Solve the pair:
$$
k_F = \frac{u_\text{up} + u_\text{down}}{2}, \qquad k_L = \frac{u_\text{up} - u_\text{down}}{2}.
$$
The <strong>average</strong> is the gravity hold; <strong>half the difference</strong> is the friction.
Two measurements give you both gains.
</Callout>

<JavaCode league="FTC" label="PDFL: P + filtered D + constant F + static-friction L (no integral)">{`@Config
public class PDFL {                       // Proportional · Derivative · Feedforward · Lower-limit
    public static double kP, kD, kF, kL;  // tune F & L from the up/down break-away powers
    public static double deadzone = 5;    // inside this band (ticks), drop the L term

    private double filteredD = 0;
    private final double alpha = 0.1;     // low-pass smoothing on the derivative
    private double lastError = 0, lastTime = 0;
    private final ElapsedTime timer = new ElapsedTime();

    /** One loop. error = target - measurement. Returns motor power. */
    public double run(double error) {
        double t  = timer.seconds();
        double dt = Math.max(1e-3, t - lastTime);
        double dError = error - lastError;
        lastError = error;  lastTime = t;

        double p = kP * error;

        double rawD = dError / dt;                          // (use -measurementRate to also kill kick)
        filteredD = alpha * rawD + (1 - alpha) * filteredD; // EMA low-pass on the derivative
        double d = kD * filteredD;

        double f = kF;                                      // constant: holds against gravity
        double l = kL * Math.signum(error);                 // static-friction floor

        // near the target, sign(error) flips each loop -> drop L so it can't buzz
        return (Math.abs(error) < deadzone) ? p + d + f : p + d + f + l;
    }
}`}</JavaCode>

This mirrors the widely shared FTC `PDFL` controller. There is no integral term at all: $k_F$ holds the
steady load and $k_L$ breaks friction, so there is nothing to wind up.

PID reacts only to error that has *already* happened. Pair it with the
[feedforward](./feedforward.mdx) — feedforward for the bulk of the command, PID for the residual it
can't predict — and you have the control stack behind nearly every fast, accurate FTC mechanism.

From here the module branches into the techniques that make that stack *good*. For a high-inertia
flywheel, a far simpler controller often beats PID — [**bang-bang**](./bang-bang.mdx) (and its
take-back-half upgrade). To
move a mechanism fast *without* overshoot, stop feeding it a step and feed it a
[**motion profile**](./motion-profiling.mdx) instead. And to keep every one of these behaving identically
from a full battery to a sagging one, wrap them all in [**voltage compensation**](./voltage-compensation.mdx).
