# From Math to a Loop

<Difficulty level="Medium" />

<Abstract>
Control theory is written in continuous time, but a robot runs it as a repeating loop over discrete
samples, so derivatives become differences, integrals become running sums, and raw encoder counts
become real-world units. This lesson bridges the math to the loop: the measured time step, unit
conversion, and the Nyquist limit on how fast you must sample.
</Abstract>

Every formula in this curriculum is written in continuous time: integrals, derivatives, smooth
curves. The hardware does none of that. This page is the bridge between the analysis and the
embedded reality, and skipping it is the single most common reason "correct" control code
misbehaves on a real robot.

<Problem title="The hub runs discretely, in the wrong units, at a variable rate">

Three hardware facts break the clean continuous math:

- **It loops.** A Control Hub executes *read -> compute -> actuate* perhaps 20-50 times per second.
  It cannot integrate or differentiate; it can only work with discrete samples.
- **It counts ticks.** A motor encoder reports dimensionless counts, not meters or radians. Nothing
  downstream, including PID gains, feedforward, and odometry, is meaningful until those counts are converted.
- **The loop time wobbles.** Vision frames, logging, and garbage collection make each iteration take
  a slightly different amount of time. Assuming a fixed period silently corrupts every time-based term.

Get these three things right and the rest of the site's math drops onto hardware unchanged. Get
them wrong and even a perfect controller drifts, lags, or oscillates.

</Problem>

<Theory title="Discretization, units, and the sampling limit">

**Time step.** Replace "an instant" with "one loop." A derivative becomes a difference; an integral
becomes a running sum; a state is advanced with forward Euler:

$$
\frac{dx}{dt} \approx \frac{x_k - x_{k-1}}{\Delta t}, \qquad
\int x\,dt \approx \sum_k x_k\,\Delta t, \qquad
x_{k+1} = x_k + \dot{x}_k\,\Delta t
$$

Every term carries a $\Delta t$, so $\Delta t$ must be the *measured* elapsed time, never a constant.

**Units.** The most important conversion in the codebase turns encoder ticks into real distance:

$$
\text{meters} = \text{ticks} \times \frac{2\pi r}{N_{\text{out}}}
$$

where $r$ is the wheel radius and $N_{\text{out}}$ is encoder counts per output-shaft revolution.
If your datasheet gives encoder counts per motor revolution instead, multiply by the gearbox ratio
first:

$$
N_{\text{out}} = N_{\text{motor}}G
$$

where $N_{\text{motor}}$ is counts per motor revolution and $G$ is the gear ratio from motor shaft to output shaft.

<Callout type="math" title="Worked example (motor configuration)">
Suppose your selected motor configuration or datasheet gives <strong>537.7 encoder counts per output revolution</strong>
and the wheel diameter is <strong>96 mm</strong>, so $r = 0.048\ \text{m}$. One wheel revolution covers
<code>2*pi*0.048 ~= 0.3016 m</code>, so
<code>METERS_PER_TICK = 0.3016 / 537.7 ~= 0.000561 m/tick</code>.
</Callout>

**Sampling rate.** Between loops the robot is blind: it acts on the last sample until the next
arrives. Increase **dt** below and watch the smooth signal degrade into a coarse staircase and,
past the limit, alias into nonsense.

<Callout type="math" title="The Nyquist limit.">
To capture a motion that repeats <code>f</code> times per second, you must sample faster than
<code>2f</code>. Below that threshold the motion <em>aliases</em>: it appears slower than reality
or disappears entirely. A loop that is too slow cannot control a fast, springy mechanism regardless
of how well its gains are tuned.
</Callout>

</Theory>

<Deploy title="Measure dt; define the unit conversion once">

Read the true loop time every iteration, and centralize the tick conversion so every downstream
controller speaks real units.

<JavaCode league="FTC" label="Real loop time + a single source of units">{`// --- units: define once, use everywhere ---
static final double WHEEL_R = 0.048;          // meters
static final double TICKS_PER_OUTPUT_REV = 537.7;
static final double METERS_PER_TICK = 2 * Math.PI * WHEEL_R / TICKS_PER_OUTPUT_REV;

ElapsedTime timer = new ElapsedTime();

while (opModeIsActive()) {
    double dt = timer.seconds();
    timer.reset();
    if (dt <= 0) dt = 1e-3;                   // guard the first loop

    double meters = motor.getCurrentPosition() * METERS_PER_TICK;
    double metersPerSecond = motor.getVelocity() * METERS_PER_TICK;  // ticks/s -> m/s

    // every time-based term now uses the measured dt:
    integral += error * dt;
    derivative = (meters - lastMeters) / dt;
}`}</JavaCode>

<Callout type="warn" title="Hard-coding dt is a classic, invisible bug.">
If you assume <code>dt = 0.02</code> but the loop actually runs at 35 ms, the integral accumulates
about 1.75 times too fast and the derivative is off by the same factor, with no error message. Measuring it
costs almost nothing.
</Callout>
