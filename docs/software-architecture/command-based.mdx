# Command-Based Programming

<Difficulty level="Medium" />

<Abstract>
Command-based programming builds robot behavior from small reusable command objects, then lets a scheduler run, sequence, interrupt, and compose them.
It scales the non-blocking discipline of state machines from one subsystem to the whole robot, especially during autonomous.
</Abstract>

A state machine is excellent when one subsystem has clear modes. Command-based programming becomes
useful when the robot needs to coordinate several subsystems at once.

The key difference is composition. A command is not "the whole autonomous." A command is one reusable
action: drive to a pose, raise the lift, aim the turret, score, retract, wait until a sensor trips.
The autonomous routine is built by combining those actions.

## FTC example: the step-counter autonomous

Most teams eventually write something like this:

<JavaCode league="FTC" label="The step counter that starts simple">{`switch (step) {
    case 0:
        driveToSpikeMark();
        if (drive.atTarget()) step++;
        break;

    case 1:
        liftToScore();
        if (lift.atTarget()) step++;
        break;

    case 2:
        openClaw();
        if (timer.seconds() > 0.4) step++;
        break;
}`}</JavaCode>

This works until you need the lift to rise while the drivetrain is still moving. Now `step == 0` is
not only "drive." It is "drive while maybe lifting." Then you want to aim while lifting. Then you want
to cancel the routine if localization fails. The step counter becomes a pile of special cases.

Command-based programming changes the question from:

> "What numbered step am I on?"

to:

> "Which commands are currently scheduled, and which subsystems do they require?"

That is a better question for an FTC robot.

## Theoretical rigor

A command has a lifecycle:

| Method | Called | Purpose |
|---|---|---|
| `initialize()` | once, when scheduled | capture targets, reset timers, prepare hardware |
| `execute()` | every loop while active | do one non-blocking slice of work |
| `isFinished()` | every loop after `execute()` | report whether the command is done |
| `end(interrupted)` | once, after finish or cancellation | stop motors, hold position, clean up |

Let $C_k$ be the set of commands scheduled at loop $k$. The scheduler runs each active command once:

$$
\forall c \in C_k:
\quad
c.\texttt{execute}()
\quad\text{then check}\quad
c.\texttt{isFinished}()
$$

Here, $c$ is one command object, and $C_k$ is the active command set during the current robot loop.

The scheduler also enforces subsystem requirements. If command $a$ and command $b$ both require the
same subsystem, they cannot safely run together:

$$
\operatorname{req}(a) \cap \operatorname{req}(b) \ne \varnothing
$$

Here, $\operatorname{req}(a)$ means "the set of subsystems required by command $a$." If the
intersection is not empty, both commands want at least one of the same subsystems.

That one rule prevents two commands from writing conflicting motor outputs to the same mechanism in
the same loop.

### Groups are commands too

The powerful part is that command groups share the same lifecycle as ordinary commands.

| Group | Behavior | FTC use case |
|---|---|---|
| Sequential | run command A, then B, then C | drive, score, park |
| Parallel | run A and B at the same time | drive while raising lift |
| Race/deadline | end a group when one condition finishes | drive until pose reached or timeout |

Because groups are commands, groups can contain groups. That is how a full autonomous routine becomes
readable without becoming one huge class.

## Enterprise implementation

A minimal command interface is small. The complexity belongs in the scheduler and the reusable command
objects, not in a giant OpMode.

<JavaCode league="FTC" label="Command interface">{`public interface Command {
    default void initialize() {}

    /**
     * Runs one loop of work. This method must return quickly.
     */
    default void execute() {}

    /**
     * @return true when this command has completed its goal.
     */
    default boolean isFinished() {
        return false;
    }

    /**
     * @param interrupted true if another command or cancellation stopped this command.
     */
    default void end(boolean interrupted) {}

    /**
     * @return subsystems this command needs exclusive control over.
     */
    Set<Subsystem> getRequirements();
}`}</JavaCode>

A command should do one job and expose its requirement clearly:

<JavaCode league="FTC" label="Reusable lift command">{`public final class RaiseLift implements Command {
    private final Lift lift;
    private final double targetInches;

    public RaiseLift(Lift lift, double targetInches) {
        this.lift = lift;
        this.targetInches = targetInches;
    }

    @Override
    public void initialize() {
        lift.setTargetHeight(targetInches);
    }

    @Override
    public void execute() {
        lift.updateController();
    }

    @Override
    public boolean isFinished() {
        return lift.atTarget();
    }

    @Override
    public void end(boolean interrupted) {
        lift.holdCurrentPosition();
    }

    @Override
    public Set<Subsystem> getRequirements() {
        return Set.of(lift);
    }
}`}</JavaCode>

Now autonomous can be written as composition:

<JavaCode league="FTC" label="Autonomous as composed commands">{`Command scorePreload = new SequentialCommandGroup(
        new ParallelCommandGroup(
                new DriveToPose(drive, scoringPose),
                new RaiseLift(lift, HIGH_JUNCTION)),
        new ScoreGameElement(claw),
        new ParallelCommandGroup(
                new RetractLift(lift),
                new DriveToPose(drive, parkPose)));

scheduler.schedule(scorePreload);

while (opModeIsActive()) {
    drive.updatePoseEstimate();
    scheduler.run();
    telemetry.update();
}`}</JavaCode>

That routine says what the robot should do without hiding the timing inside a pile of step numbers.
Drive and lift overlap because they require different subsystems. Scoring waits because it comes after
the parallel group.

## FTC command frameworks worth knowing

You can write a scheduler yourself, but most serious teams should at least study the existing FTC
frameworks. They encode the same architecture this lesson has been building: commands, subsystem
requirements, command groups, trigger bindings, and a scheduler that runs every loop.

This is not a ranking. It is a map of tradeoffs.

| Framework | Best fit | Main idea | Cost |
|---|---|---|---|
| [FTCLib](https://docs.ftclib.org/ftclib/command-base/command-system/command-scheduler) | Understanding the original FTC command-based ecosystem | WPILib-style commands and scheduler ported into FTC | Older baseline; many teams now prefer maintained forks |
| [SolversLib](https://docs.seattlesolvers.com/) | Teams that want FTCLib-style command-based with active maintenance | Updated maintained fork of older FTCLib, designed to avoid breaking most existing FTCLib code | Still inherits the WPILib/FTCLib mental model |
| [Mercurial](https://docs.dairy.foundation/Mercurial/overview) | Experienced teams that want powerful composition and better developer ergonomics | Lambda-first command builders, command groups, run states, requirements, interruptibility, and stronger diagnostics | Harder first setup and a more abstract API |
| [Ivy](https://pedropathing.com/docs/ivy) | Pedro Pathing teams that want command-based autonomous without much ceremony | Lightweight command-based control flow library built by the Pedro Pathing ecosystem | Newer, smaller ecosystem than FTCLib-style frameworks |

### FTCLib: the historical FTC command baseline

FTCLib is the vocabulary source for many FTC command-based projects: `Command`, `Subsystem`,
`CommandScheduler`, command groups, default commands, and subsystem requirements. Its scheduler docs
are still useful for learning the loop sequence, but for new robots treat it as the older baseline
and compare it against SolversLib before committing.

### SolversLib: maintained FTCLib-style command-based

[SolversLib](https://docs.seattlesolvers.com/) describes itself as an updated and maintained fork of
older FTCLib, designed to preserve most existing FTCLib code. Use it when you want the familiar
WPILib/FTCLib architecture with current maintenance: `CommandOpMode`, command groups, requirements,
and default commands without asking the team to learn a more abstract framework.

### Mercurial: powerful composition for advanced teams

[Mercurial](https://docs.dairy.foundation/Mercurial/overview) is also command-based, but it is not a
simple FTCLib clone. It pushes teams toward `Lambda` and `StatefulLambda` builders, command utilities
like `Parallel`, `Sequential`, `Race`, `Wait`, and `IfElse`, plus stronger error messages and a
unit-tested scheduler. Use it when your programmers are already comfortable with lambdas,
requirements, interruptibility, and builder-style APIs; otherwise the abstraction can become noise.

### Ivy: command-based control flow for Pedro Pathing

[Ivy](https://pedropathing.com/docs/ivy) is Pedro Pathing's command-based control flow library. Pedro's
own autonomous examples recommend Ivy for composing `follow` commands with subsystem actions. If your
robot already uses Pedro, Ivy is the low-friction way to write autos as `sequential` and `parallel`
command compositions instead of manually managing path state.

Use Ivy when your main architecture question is:

> "How do I make Pedro paths and mechanism actions compose cleanly?"

Use SolversLib or Mercurial when your main architecture question is broader:

> "How do I organize the whole robot's teleop, autonomous, triggers, and subsystems around commands?"

<Admonition type="tip" title="Framework choice is an architecture decision">

Pick the framework whose complexity matches the team's maturity. A simple scheduler used correctly is
better than an advanced framework used as a maze. The goal is not to collect abstractions; the goal is
to make robot behavior composable, inspectable, and hard to accidentally break.

</Admonition>

## Hardware reality

<Admonition type="warning" title="Requirements are not optional">

The scheduler can only prevent command conflicts if commands declare their subsystem requirements. If
two commands both write to the drivetrain but one forgot to require `drive`, the scheduler cannot save
you.

</Admonition>

<Admonition type="warning" title="Commands must be non-blocking">

`execute()` should do one loop of work and return. Do not put `sleep()`, long `while` loops, or
blocking sensor waits inside a command. Use `isFinished()` to express completion conditions.

</Admonition>

<Admonition type="tip" title="State machines and commands combine cleanly">

A subsystem can use a state machine internally and still expose commands externally. For example, an
intake may have `IDLE`, `INTAKING`, and `JAM_CLEARING` states, while autonomous schedules simple
commands like `RunIntakeUntilLoaded`.

</Admonition>

## Choosing the architecture

Use a state machine when one subsystem has a few exclusive modes. Use command-based programming when
you need to sequence, parallelize, cancel, and reuse behaviors across multiple subsystems.

In practice, strong FTC code often uses both:

- state machines inside complex subsystems,
- commands for autonomous and operator actions,
- measured loop time and telemetry everywhere.
