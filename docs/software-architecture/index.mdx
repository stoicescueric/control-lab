# Module 1 · Software Architecture & Loop Optimization

Control theory does not run in a vacuum. It runs inside an OpMode, inside a loop, on hardware with
limited time and imperfect sensors.

This module answers two questions before the math gets heavier:

1. Where does robot behavior live in the codebase?
2. What does it mean for control math to run once per loop on a real Control Hub?

## What this module covers

- **State machines** - model one subsystem as explicit modes and legal transitions.
- **Command-based programming** - compose reusable actions across the whole robot.
- **Discretization and units** - turn continuous equations into per-loop Java using measured
  $\Delta t$ and real engineering units.
- **Loop optimization** - protect the loop-time budget that filters, controllers, and estimators all
  depend on.

<Admonition type="tip" title="Why architecture comes before control theory">

The most common reason good control math fails in competition is not the equation. It is that the
equation was dropped into a giant OpMode where timing, units, telemetry, and subsystem ownership were
unclear. Architecture makes the rest of this curriculum deployable.

</Admonition>
