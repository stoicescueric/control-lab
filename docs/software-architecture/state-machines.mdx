# State Machines

<Abstract>
A finite state machine models a subsystem as one current mode plus a small set of legal transitions.
For FTC robots, that makes sequencing code easier to debug because impossible combinations, like intaking and scoring at the same time, become impossible to represent.
</Abstract>

FTC code fails in boring ways before it fails in mathematically interesting ways. A good PID controller
does not save an arm routine that is half blocked by `sleep()`, half controlled by stale booleans, and
impossible to reason about from telemetry.

A state machine is the first architecture tool that gives robot behavior a shape.

## FTC example: the boolean-flag scorer

A scoring subsystem often starts clean:

<JavaCode league="FTC" label="The innocent beginning">{`boolean isIntaking = false;
boolean hasSample = false;
boolean isLifting = false;
boolean isScoring = false;`}</JavaCode>

Then the robot gets more capable. The driver wants cancel. The intake has a beam-break sensor. The arm
needs time to move. The claw needs to release after the arm reaches the top. Suddenly the robot has
many flags, and the flags can disagree.

What does this mean?

<JavaCode league="FTC" label="Contradictory subsystem state">{`isIntaking = true;
hasSample = true;
isLifting = true;
isScoring = false;`}</JavaCode>

Maybe it means the robot is still pulling in a game element while the lift is already moving. Maybe it
means the flags were not reset after the last cycle. Maybe it means the next `if` block will do
something dangerous. The code can enter states the robot should never physically enter.

The bug is not just "too many booleans." The bug is that the subsystem does not have one authoritative
answer to the question: **what mode am I in right now?**

## Theoretical rigor

A finite state machine has three pieces:

| Symbol | Meaning | FTC scorer example |
|---|---|---|
| $S$ | set of states | `IDLE`, `INTAKING`, `LOADED`, `RAISING`, `SCORING` |
| $E$ | set of events | driver button, sample detected, arm at target |
| $\delta$ | transition function | rule that maps state plus event to next state |

The transition function is:

$$
\delta : S \times E \rightharpoonup S
$$

Here, $S \times E$ means "a state paired with an event." The half-arrow $\rightharpoonup$ means the
function is partial: not every event is legal from every state.

A small scorer can be written as a transition table:

| Current state | Event | Next state |
|---|---|---|
| `IDLE` | driver starts intake | `INTAKING` |
| `INTAKING` | sample detected | `LOADED` |
| `INTAKING` | driver cancels | `IDLE` |
| `LOADED` | driver requests score | `RAISING` |
| `RAISING` | arm reaches target | `SCORING` |
| `SCORING` | release timer expires | `IDLE` |

Everything not listed is ignored. That is not laziness; it is the safety property. The machine cannot
be `INTAKING` and `RAISING` simultaneously because `state` has exactly one value.

State machines also make outputs easier to audit. In a simple Moore-style machine, hardware behavior
comes from the current state:

$$
u_k = g(s_k)
$$

Here, $s_k$ is the current state at loop $k$, and $u_k$ is the actuator command chosen for that state.
For example, `INTAKING` runs the intake, `RAISING` commands the arm target, and `IDLE` stops or holds
everything safely.

recoverable robot mode, and every arrow represents a condition that can be checked during the loop.

## Enterprise implementation

The deployable FTC pattern is an `enum`, a `switch`, and a timer, but a real autonomous machine
needs more than a pretty transition table. It needs **entry actions**, **timed exits**,
**path-completion guards**, **interference recovery**, and **subsystem updates** that still run every
loop.

The excerpt below is an adapted, teaching-oriented reconstruction of the WolfPackMachina
`Auto.java` pattern. It is not the full file. The original uses `Paths.Path` objects directly as
states; this version wraps the same idea in a smaller `AutoState` enum so the architecture is easier
to read.

<JavaCode league="FTC" label="Adaptive autonomous FSM — adapted from WolfPackMachina Power Play Auto.java">{`public abstract class AdaptiveAuto {
    private enum AutoState {
        WAIT_FOR_PICKUP,
        DISLODGE_CONE,
        ALIGN_LEFT_RAM,
        LEFT_RAM,
        CONTESTED_LEFT_RAM,
        ALIGN_CONTESTED,
        ALIGN_PARK,
        PARK_CLOSE,
        CYCLE_TRANSFER
    }

    private AutoState currentState = AutoState.WAIT_FOR_PICKUP;
    private AutoState scoringState = AutoState.ALIGN_CONTESTED;

    private final ElapsedTime stateTimer = new ElapsedTime();
    private final ElapsedTime settleTimer = new ElapsedTime();

    private final Movement drive;
    private final ClawIntake intake;
    private final Depositor depositor;

    private boolean pathComplete = false;
    private boolean parking = false;
    private boolean contracting = true;
    private boolean robotSettled = false;
    private double slidesPositionAtEntry = 0.0;

    /**
     * Runs exactly one autonomous loop. This method must never block.
     */
    public final void loop(Telemetry telemetry) {
        switch (currentState) {
            case ALIGN_PARK:
                alignPark();
                break;

            case DISLODGE_CONE:
                dislodgeCone();
                break;

            case ALIGN_LEFT_RAM:
                alignLeftRam();
                break;

            case LEFT_RAM:
                runLeftRam();
                break;

            case CONTESTED_LEFT_RAM:
                recoverFromContestedLeftRam();
                break;

            case WAIT_FOR_PICKUP:
                holdScoringPosition();
                break;

            case PARK_CLOSE:
            case ALIGN_CONTESTED:
            case CYCLE_TRANSFER:
                runPathState(currentState);
                break;
        }

        updateSubsystems();
        telemetry.addData("auto/state", currentState);
        telemetry.addData("auto/scoringState", scoringState);
        telemetry.addData("auto/timeInStateMs", stateTimer.milliseconds());
        telemetry.addData("auto/pathComplete", pathComplete);
    }

    /**
     * All state changes go through one gate so entry timing and local flags reset together.
     */
    private void transitionTo(AutoState nextState) {
        if (nextState == currentState && nextState != AutoState.CYCLE_TRANSFER) {
            return;
        }

        currentState = nextState;
        pathComplete = false;
        stateTimer.reset();
        settleTimer.reset();
    }

    /**
     * A driver-independent request can redirect the autonomous routine into parking.
     */
    public final void requestPark() {
        if (currentState != AutoState.ALIGN_PARK) {
            transitionTo(AutoState.ALIGN_PARK);
        }
    }

    /**
     * First-entry logic selects a park target once; later loops only wait for a safe transition.
     */
    private void alignPark() {
        if (!parking) {
            Pose2d target = getClosestParkingSquare();
            setParkingTarget(target);
            parking = true;
            intake.retractSlides();
        }

        drive.stopDrive();

        if (stateTimer.milliseconds() > 650) {
            transitionTo(AutoState.PARK_CLOSE);
        }
    }

    /**
     * Recovery state: push a stuck cone free, then return to the pickup cycle.
     */
    private void dislodgeCone() {
        if (!pathComplete) {
            depositor.setV4BState(Depositor.V4BState.HOME);
            slidesPositionAtEntry = intake.getSlidesPosition();
            intake.retractSlides();

            pathComplete = true;
            stateTimer.reset();
            return;
        }

        drive.holdPosition(getScoreHoldPose());

        if (stateTimer.milliseconds() > 950
                && (slidesPositionAtEntry <= intake.getContestedHighExtension()
                || stateTimer.milliseconds() > 1400)) {
            intake.retractSlides();
            transitionTo(AutoState.WAIT_FOR_PICKUP);
            return;
        }

        double slidesTarget = slidesPositionAtEntry + 280.0;
        intake.setSlidesPosition(slidesTarget, 0.6);

        if (stateTimer.milliseconds() < 700) {
            Pose2d pushPose = getScoreHoldPose().translatedForward(28.0);
            drive.holdPosition(pushPose);
        }
    }

    /**
     * Align before ramming into the stack. The robot contracts first so mechanisms are safe.
     */
    private void alignLeftRam() {
        if (contracting && depositor.getLiftPosition() > depositor.getIntakeThreshold() * 3.0) {
            drive.holdPosition(getScoreHoldPose());
            return;
        }

        contracting = false;
        contractRobot(true, true);

        double headingHold = getLeftRamPath().progress() < 0.15
                ? getScoreHoldPose().heading
                : 0.0;

        boolean stillFollowing = drive.followPath(
                getLeftRamAlignmentPath(),
                0.75,
                headingHold,
                MovementDash.FAST_VELOCITY_THRESHOLD);

        if (!stillFollowing) {
            drive.shiftLocation(0.0, 0.0);
            transitionTo(AutoState.LEFT_RAM);
        }
    }

    /**
     * Main pickup path. If field interference moved the robot off-line, redirect to recovery.
     */
    private void runLeftRam() {
        if (getLeftRamPath().progress() > 0.05) {
            intake.setSlidesTarget(AutoDashboard.RAM_INTAKE_EXTENSION);
        }

        boolean stillFollowing = drive.followPath(
                getLeftRamPath(),
                1.0,
                0.0,
                MovementDash.FAST_VELOCITY_THRESHOLD / 3.0);

        if (stillFollowing) {
            return;
        }

        intake.retractSlides();

        if (scoringState != AutoState.ALIGN_CONTESTED) {
            scoringState = AutoState.ALIGN_CONTESTED;
            switchYawScoreDirection();
            setCloseHighScorePosition();
        }

        if (robotWasPushedOffLine()) {
            transitionTo(AutoState.CONTESTED_LEFT_RAM);
        } else {
            transitionTo(AutoState.ALIGN_CONTESTED);
        }

        getLeftRamPath().reset();
    }

    /**
     * Fail-safe path after an interaction with defense or field contact.
     */
    private void recoverFromContestedLeftRam() {
        boolean stillFollowing = drive.followPath(
                getContestedLeftRamPath(),
                0.75,
                0.0,
                MovementDash.FAST_VELOCITY_THRESHOLD);

        if (!stillFollowing) {
            transitionTo(AutoState.LEFT_RAM);
        }
    }

    /**
     * Hold the scoring pose unless the robot is not settled or has been pushed off-line.
     */
    private void holdScoringPosition() {
        Pose2d scorePose = getScoreHoldPose();
        robotSettled = drive.holdPosition(scorePose);

        if (!robotSettled || robotWasPushedOffLine()) {
            contractRobot(currentState != AutoState.WAIT_FOR_PICKUP, true);
            return;
        }

        settleTimer.reset();
    }

    private void updateSubsystems() {
        drive.update();
        intake.update();
        intake.stateMachine();
        depositor.update();
        depositor.stateMachine();
    }

    protected abstract Pose2d getClosestParkingSquare();
    protected abstract Pose2d getScoreHoldPose();
    protected abstract Path getLeftRamPath();
    protected abstract Path getLeftRamAlignmentPath();
    protected abstract Path getContestedLeftRamPath();
}`}</JavaCode>

This is what makes the example valuable. The state machine is not just replacing `sleep()` with
`if`; it is creating a recoverable autonomous policy. `ALIGN_LEFT_RAM` can refuse to move until the
robot is contracted. `LEFT_RAM` can branch into `CONTESTED_LEFT_RAM` if the robot is pushed off-line.
`ALIGN_PARK` has first-entry logic so the parking target is chosen once, not recalculated every loop.
Every transition resets local timing so `stateTimer.milliseconds()` always means "time since this
state began."

In mathematical language, this is still the same partial transition function:

$$
\delta(s_k, e_k) = s_{k+1}
$$

but now the event $e_k$ is not only a button press. It can be "path finished," "650 ms elapsed,"
"robot was interfered with," "slides are below a safe threshold," or "parking has been requested."
That is why autonomous FSMs are so powerful: they turn a linear script into a policy that can react
to the field.

## Hardware reality

<Admonition type="warning" title="Never block the OpMode loop">

`sleep()` and long `while` waits freeze the entire robot. Driver control, sensor updates, telemetry,
and other subsystems all stop. A state machine replaces "wait here" with "check whether the condition
is true this loop."

</Admonition>

<Admonition type="tip" title="Telemetry should name the state">

Print the current state and time-in-state every loop. During a match-debug session, "stuck in
RAISING for 2.4 seconds" is vastly more useful than staring at four unrelated booleans.

</Admonition>

<Admonition type="tip" title="Use state machines inside subsystems">

State machines are excellent for one mechanism with a few clear modes: intake, shooter, lift, claw,
endgame latch. When the problem becomes composing many subsystems into a full autonomous routine, use
[Command-Based Programming](./command-based.mdx) on top of those subsystem internals.

</Admonition>
