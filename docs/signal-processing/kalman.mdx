# Kalman Filter

<Difficulty level="Hard" />

<Abstract>
The Kalman filter is the optimal linear state estimator under the usual linear-Gaussian assumptions:
it predicts the state from a motion model, then corrects with each measurement in the proportion that
minimises variance, all while tracking its own uncertainty. This lesson derives that predict-correct
cycle and the gain that decides how far to trust the model versus the measurement.
</Abstract>

The Kalman filter is the optimal linear state estimator when the model is linear, the noise is
Gaussian, and the covariances are honest: given a motion model and noisy measurements, it produces
the minimum-variance estimate of the true state, and tracks its own uncertainty while doing so. It is
the principled successor to the hand-tuned complementary filter.

Kou and Haggenmiller's FTC-focused paper,
[*Extended Kalman Filter State Estimation for Autonomous Competition Robots*](https://arxiv.org/abs/2310.04459),
is the applied reference for this lesson. Their competition-robot setup is exactly the FTC pattern
we care about: odometry gives frequent motion information, computer vision landmarks give less
frequent absolute position information, and the filter fuses them by comparing their covariance.

<Problem title="A fixed blend is not the optimal blend">

The complementary filter blends two sources with a *constant* weight $\alpha$. But the right amount
to trust a measurement is not constant — it depends on how uncertain your prediction currently is
versus how noisy the sensor is. On a robot this is concrete: dead-reckoning odometry is smooth but
drifts, while an absolute fix (an AprilTag pose, a beacon) is noisy and intermittent. Early after a
correction the model is trustworthy; after a long blind stretch it is not. A fixed weight cannot
express that, and it cannot report how confident the resulting estimate is.

</Problem>

<Theory title="Predict, update, and the optimal gain">

The filter maintains a state estimate $\hat{x}$ and a covariance $P$ (its uncertainty), and repeats
two steps. **Predict** advances the state through the motion model and *grows* the uncertainty:

$$
\hat{x}^- = A\hat{x} + Bu, \qquad P^- = A P A^\top + Q
$$

<EquationLegend
  title="Prediction symbols"
  items={[
    {symbol: 'x-hat', meaning: 'the filter estimate of the state, such as position or velocity'},
    {symbol: 'A', meaning: 'state-transition matrix: how the state naturally advances one loop'},
    {symbol: 'B', meaning: 'input matrix: how the control command changes the state'},
    {symbol: 'u', meaning: 'control input, such as commanded velocity or acceleration'},
    {symbol: 'P', meaning: 'state covariance: uncertainty attached to the estimate'},
    {symbol: 'Q', meaning: 'process-noise covariance: uncertainty added because the model is imperfect'},
  ]}
/>

Here $\hat{x}$ is the current best estimate of the state, such as position or velocity; $A$ is the
state-transition matrix that advances the state by one loop; $u$ is the control input; $B$ maps that
control input into state change; $P$ is the covariance matrix, meaning the uncertainty attached to
the estimate; and $Q$ is process noise, the uncertainty added because the model is never perfect.
The paper writes this as the prediction model $x_{a,t}=Ax_{a,t-1}+Bu_t+q_t$, where $q_t$ is
Gaussian process noise with covariance $Q$.

If the state has only one number, such as "robot x-position," covariance reduces to ordinary
variance. In that scalar case the update gain becomes:

$$
K = \frac{P^-}{P^- + R}
$$

That formula is the whole intuition. If $P^-$ is large, the prediction is uncertain, so $K$ moves
toward $1$ and the measurement gets trusted. If $R$ is large, the sensor is noisy, so $K$ moves
toward $0$ and the prediction stays mostly unchanged. The matrix equation is the same idea, except
the filter can track uncertainty across several coupled state variables at once.

**Update** folds in a measurement $z$ and *shrinks* the uncertainty, weighted by the **Kalman
gain** $K$:

$$
K = P^- H^\top (H P^- H^\top + R)^{-1}, \qquad
\hat{x} = \hat{x}^- + K\,(z - H\hat{x}^-), \qquad
P = (I - KH)P^-
$$

<EquationLegend
  title="Update symbols"
  items={[
    {symbol: 'z', meaning: 'sensor measurement received this loop'},
    {symbol: 'H', meaning: 'measurement matrix: maps state space into sensor space'},
    {symbol: 'R', meaning: 'measurement-noise covariance: how noisy the sensor is'},
    {symbol: 'K', meaning: 'Kalman gain: how strongly the innovation should correct the prediction'},
    {symbol: 'I', meaning: 'identity matrix, used to shrink covariance after the correction'},
  ]}
/>

In this update, $z$ is the sensor measurement, $H$ maps the state into whatever the sensor can
observe, $R$ is measurement-noise covariance, and $I$ is the identity matrix. The term
$(z-H\hat{x}^-)$ is the innovation: the part of the measurement the model did not already predict.

Read the update in three pieces:

$$
\text{innovation} = z - H\hat{x}^-
$$

$$
\text{correction} = K \cdot \text{innovation}
$$

$$
\text{new estimate} = \text{prediction} + \text{correction}
$$

The innovation answers, "How surprised was the filter by this sensor reading?" The Kalman gain
answers, "How much of that surprise should we believe?" The covariance update answers, "After
seeing this measurement, how much uncertainty remains?"

The update is the low-pass formula you already know — *estimate = prediction + K·(measurement −
prediction)* — except $K$ is **derived optimally** from the prediction covariance $P^-$ and the
measurement noise $R$, every step:

- noisy sensor (large $R$) → small $K$ → trust the model, stay smooth;
- shaky prediction (large $P^-$) → large $K$ → jump toward the measurement.

In the demo, a target follows a smooth path reported only as noisy blips. The filter recovers a
smooth estimate plus an uncertainty bubble. The two tuning parameters are $R$
(measurement noise) and $Q$ (process noise / "model trust").

<Kalman />

<Callout type="warn" title="Q and R are the tuning.">
Set model trust very low and the estimate lags on sharp turns — it over-believes its motion model.
Set it high and it chases every blip. Choosing <code>Q</code> and <code>R</code> to match the real
process and sensor noise is the engineering of applying a Kalman filter.
</Callout>

</Theory>

<Deploy title="On a robot: fuse odometry with an absolute fix">

Full matrix Kalman filters on FTC are usually provided by a library (and WPILib ships
`SwerveDrivePoseEstimator` / `DifferentialDrivePoseEstimator` for exactly this odometry + vision
fusion). The mechanism is approachable, though — a **scalar** Kalman update is enough to fuse two
estimates of one quantity, and shows the whole idea:

<JavaCode league="FTC" label="Scalar Kalman update — fuse odometry x with a tag x">{`double x = odoX;          // current estimate (from odometry)
double P = pVar;          // its variance, grown each loop by process noise Q
// P += Q;  // (prediction step: x advances with motion, P grows)

// When an AprilTag gives an absolute x with measurement variance R:
double R = tagVar;
double K = P / (P + R);          // optimal gain
x = x + K * (tagX - x);          // corrected estimate
P = (1 - K) * P;                 // reduced uncertainty`}</JavaCode>

<Callout type="tip" title="It extracts signal from noise.">
In the demo the Kalman error is typically a fraction of the raw blip error — genuine information
recovery, not just smoothing. That is why it underpins GPS, drone, and robot localisation.
</Callout>

The basic filter assumes the motion and measurement models are **linear**. Robot localisation is
not — range/bearing and heading involve trigonometry. The Extended Kalman Filter handles that, next.

</Deploy>
