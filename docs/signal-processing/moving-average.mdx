# Moving Average

<Difficulty level="Easy" />

<Abstract>
A moving average smooths a signal by averaging the last N readings equally — excellent at suppressing
random noise, but sluggish to react and easily dragged off by a single outlier. This lesson examines
both failure modes and uses them to motivate the smarter estimator that follows.
</Abstract>

The moving average is the filter you already use intuitively — a "7-day average," a "50-day stock
average." It is excellent at one job and instructively bad at another, which sets up why a smarter
estimator is eventually required.

<Problem title="Steady jitter needs averaging; bursts and lag do not forgive it">

A signal buried in steady random jitter is best cleaned by averaging — but a simple average over *all*
history can never track a moving signal. The fix is a **sliding window**: average only the most
recent N readings, discarding the oldest as each new one arrives. The window size is a direct
smooth-versus-responsive trade-off, and — as the demo will show — a single outlier contaminates the
output for the full N samples.

<Analogy>
<strong>You have seen this.</strong> A rolling case count, a moving stock average, your phone's
average step count — all sliding-window averages that turn a spiky series into a readable trend.
</Analogy>

</Problem>

<Theory title="The average of a sliding window">

Hold the last N readings; report their mean. Next loop, the window slides one step forward.

$$
\hat{x}_k = \frac{1}{N}\sum_{i=0}^{N-1} z_{k-i}
$$

It is a close cousin of the low-pass filter, with a different memory model:

| | Moving average | Low-pass (EMA) |
|---|---|---|
| Remembers | last N readings, **equally** | all readings, **fading** |
| Storage | N values | one number |
| Setting | window size N | smoothing α |
| Best at | killing steady jitter | cheap real-time smoothing |

The demo overlays truth, sensor, and the windowed average. Grow N for smoothness, then run the step
test to feel the lag a large window costs.

<MovingAverage />

<Callout type="warn" title="The two failure modes.">
With a large window the step test ramps up slowly instead of snapping — pure lag. And a single wild
spike shifts the average for the full N samples. For fast-moving or spike-prone data, the Kalman
filter (two lessons ahead) is dramatically better.
</Callout>

</Theory>

<Deploy title="A fixed-cost ring buffer">

On a Control Hub, implement the window as a ring buffer with a running sum, so each update is O(1)
regardless of N. The cost to weigh against the low-pass is the N-value memory.

<JavaCode league="FTC" label="O(1) moving average">{`final int N = 12;
double[] buf = new double[N];
int idx = 0, count = 0;
double sum = 0;

double update(double reading) {
    sum -= buf[idx];          // drop the oldest sample in this slot
    buf[idx] = reading;        // store the newest
    sum += reading;
    idx = (idx + 1) % N;
    count = Math.min(count + 1, N);
    return sum / count;        // average of the current window
}`}</JavaCode>

Both filters here assume a *single* noisy sensor. The next two lessons fuse *multiple* imperfect
sensors into one estimate that beats any of them alone — starting with the complementary filter.

</Deploy>
