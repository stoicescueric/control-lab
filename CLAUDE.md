# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Control Lab is an interactive FTC-robotics control-theory curriculum built on **Docusaurus v3 + React 19 + TypeScript**. Content is MDX lessons in `docs/`; the "product" is the rendered static site deployed to GitHub Pages. There is no backend.

## Commands

```bash
npm start            # dev server at http://localhost:3000/control-lab/ (note the /control-lab/ baseUrl)
npm run build        # production build into build/ — this is the deployment gate and the CI check
npm run serve        # preview the production build
npm run clear        # clear Docusaurus cache (use when the sidebar/config seems stale)
npm run typecheck    # tsc — see caveat below
npm test             # vitest run (one-shot)
npm run test:watch   # vitest watch
npx vitest run src/lib/controlMath.test.ts   # run a single test file
npm run size         # bundle-size report (scripts/bundle-size.mjs)
```

### Build/typecheck reality

- **`npm run build` is the real gate.** CI (`.github/workflows/ci.yml`) runs `npm run build` only, on PRs and non-main branches. Because `onBrokenLinks: 'throw'` in `docusaurus.config.ts`, a green build also means no broken internal links. Pushes to `main` are built+deployed by `deploy.yml`.
- **`npm run typecheck` currently fails on pre-existing debt** — the `usePlot`/canvas ref typing in the older `.js` lib helpers (ref typed `null`, so `if (p)` narrows to `never`). Do not treat these existing errors as regressions. Keep any *new* TypeScript you add cleanly typed; fix typecheck findings deliberately rather than reflexively.

## Architecture

### Content pipeline (the important big picture)

1. **Lessons are `.mdx` files under `docs/`.** Each module is a folder with a `_category_.json` (`label` + `position`). Within a module, page order comes from each file's `sidebar_position` frontmatter.
2. **The sidebar is fully autogenerated** (`sidebars.ts` = `{type: 'autogenerated', dirName: '.'}`). Adding a lesson = drop an `.mdx` into the right module folder + set frontmatter. Do not hand-maintain a sidebar list.
3. **Global MDX components** are registered in `src/theme/MDXComponents.tsx`. Anything exported there is usable as a bare tag in any lesson **without an import** (e.g. `<Abstract>`, `<Callout>`, `<Problem>/<Theory>/<Deploy>`, `<JavaCode>`, `<Formula>`, and the `<Demo>` kit). When you add a reusable kit component, register it here.
4. **Per-lesson simulations are imported at the top of their own `.mdx`** — they are *not* global. They live in `src/components/sims/`.

### Component layers

- `src/components/kit/` — reusable lesson primitives (layout/prose/demo shell). `Demo.tsx` exports the demo chrome (`Demo`, `Stage`, `Controls`, `Buttons`, `Button`, `Readout`, `Legend`, `DemoCanvas`); use it so every interactive widget shares controls styling.
- `src/components/sims/` — one standalone `.tsx` per interactive widget (Kalman, EKF, PurePursuit, MecanumDrive, etc.). Keep browser-only APIs (canvas, window) inside `useEffect` or event handlers so SSR build succeeds.
- `src/lib/` — pure, framework-free helpers shared across sims:
  - `.ts` files are typed and unit-tested (`controlMath.ts`, `projectile.ts` with `*.test.ts`). **`projectile.ts` is the single source of truth for the Module 6 physics model** — every Module 6 sim draws on it so the demos can't disagree.
  - `.js` files (`plot.js`, `canvas.js`, `linalg.js`) are ported-from-legacy imperative helpers with `.d.ts` shims. `canvas.js` provides the `useDprCanvas`/`usePlot` hooks (HiDPI sizing + `requestAnimationFrame` loop + self-resizing `Plot`) that sims build on.

### Styling

Tailwind CSS **v4** is layered over Docusaurus/Infima via a custom PostCSS plugin defined inline in `docusaurus.config.ts` (`tailwindPlugin`). Design tokens and theme overrides live in `src/css/custom.css`. Demos must stay readable in both light and dark themes.

### Deployment config

`ORG` and `REPO` constants at the top of `docusaurus.config.ts` drive `url`, `baseUrl`, `organizationName`, `projectName`. Changing the repo name / owner means updating these.

## Lesson authoring standard (enforced by review, see CONTRIBUTING.md)

Every substantive lesson follows this sequence: **(1)** frontmatter + `<Abstract>`; **(2)** a concrete FTC/engineering hook (a real failure, not "math is important"); **(3)** visual intuition — a demo/diagram *before* the derivation; **(4)** KaTeX math (`$...$` / `$$...$$`) with every non-obvious variable explained near the equation; **(5)** clean FTC Java (no blocking sleeps, clear telemetry) via `<JavaCode>`; **(6)** hardware-reality notes using native Docusaurus admonitions (`:::warning`, `:::tip`).

Design direction: serious, minimal, high-contrast technical documentation. No gamer/hacker aesthetic, neon, or decorative clutter.
