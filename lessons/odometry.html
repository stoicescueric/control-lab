Odometry & Pose Tracking

Add up a thousand tiny movements and you know where the robot is on the field — for a while.

Autonomous routines need to know the robot's pose: its position (x, y) and which way it's facing, heading. No single sensor reports "you are here." So we build it up ourselves: every loop we ask the encoders how far did each wheel roll?, work out the little step the robot took, and add it onto a running total. This adding-up is called odometry (or dead reckoning).

🚶 Eyes closed. Stand up, close your eyes, and take 10 careful steps while turning. You can roughly track where you are by remembering each step. That's odometry — and just like you, it slowly gets lost, because tiny errors in each step pile up. The trick is making each step's math as honest as possible.

One odometry step

For a tank-style robot, each loop we read how far the left and right sides rolled, dL and dR (in metres — thank you, Lesson 11). From those two numbers:

forward step: ds = (dL + dR) / 2
heading change: dθ = (dR − dL) / T  (T = track width)

The simplest thing is to assume the robot went in a straight line for that step: x += ds·cos(θ), y += ds·sin(θ). That's fine in a straight line but wrong on curves — it "cuts the corner." The honest version assumes the step was a little arc (constant curvature), which is what really happens when the two sides roll different amounts. It's a few more lines and dramatically more accurate over a real path:

FTCFRC the accurate per-loop update (works in any language)
double dL = leftPos  - lastLeft;     // metres rolled since last loop
double dR = rightPos - lastRight;
lastLeft = leftPos;  lastRight = rightPos;

double ds  = (dL + dR) / 2.0;            // forward distance
double dth = (dR - dL) / TRACK_WIDTH;     // heading change (radians)

// Constant-curvature ("pose exponential") integration:
double sinC, cosC;
if (Math.abs(dth) < 1e-9) {              // basically straight -> avoid divide-by-0
    sinC = 1.0;             cosC = 0.5 * dth;
} else {
    sinC = Math.sin(dth) / dth;
    cosC = (1 - Math.cos(dth)) / dth;
}
// rotate the little arc into field coordinates and add it on:
x       += ds * (sinC * Math.cos(heading) - cosC * Math.sin(heading));
y       += ds * (sinC * Math.sin(heading) + cosC * Math.cos(heading));
heading += dth;
📡
Use the gyro for heading. Encoders estimate dθ, but wheel scrub makes that the least trustworthy number. Most teams instead take heading straight from the IMU/gyro and use encoders only for distance. WPILib's odometry classes are built to accept a gyro angle for exactly this reason.

Dead wheels: odometry that ignores slip

When a drive wheel slips or spins, your encoder lies and odometry drifts. FTC's favourite fix: dead wheels (a.k.a. odometry pods) — small unpowered omni wheels on light springs that just roll along measuring true ground movement, immune to drive slip. Two parallel pods measure forward motion and heading; a third, sideways, catches strafe.

See it drift — then fix it

The green robot drives a perfect circle (the truth). The amber robot is what odometry thinks, built only from (slightly imperfect) encoder readings. Turn up the wheel slip and watch the estimate spiral away from reality — then hit "AprilTag fix" to snap it back, the way a camera looking at a field marker would.

Odometry always drifts eventually. Errors only ever add up — they never cancel. That's not a bug, it's the nature of dead reckoning. The fix is to occasionally correct it with an absolute sensor: a camera seeing an AprilTag, distance sensors against a wall, field markings. Blending that correction in optimally is… a Kalman filter (Lesson 6), which is exactly what WPILib's SwerveDrivePoseEstimator does.
FRC · WPILib odometry in three lines
var odometry = new DifferentialDriveOdometry(
        gyro.getRotation2d(),                       // starting heading
        leftEncoder.getDistance(), rightEncoder.getDistance());

@Override
public void robotPeriodic() {
    odometry.update(gyro.getRotation2d(),           // trust the gyro for heading
                    leftEncoder.getDistance(),
                    rightEncoder.getDistance());
    Pose2d pose = odometry.getPoseMeters();         // x, y, heading — ready to use
}
