Drive Kinematics

You think in "go forward and turn." The wheels think in "spin this fast." Kinematics is the translator.

When you push a joystick you're asking for three things at once: drive forward/back, strafe sideways, and turn. We bundle those into the robot's chassis speeds. But a motor only knows how fast to spin its wheel. Kinematics is the math that converts between the two:

  • Inverse kinematics — chassis speeds → individual wheel commands. You run this every teleop loop.
  • Forward kinematics — wheel speeds → how the chassis is actually moving. The basis of odometry (next lesson).

The math depends on the drivetrain. Here are the three you'll meet in FTC/FRC.

① Tank / differential drive

Wheels on each side, no sideways motion. Turning means spinning the two sides at different speeds (like a tank, or a shopping trolley you steer by dragging one side). With chassis forward speed v, turn rate ω, and track width T (distance between the sides):

left = v − ω · (T/2)
right = v + ω · (T/2)

② Mecanum drive

Four special wheels with diagonal rollers. By mixing wheel directions, a mecanum robot can drive forward, strafe sideways, and turn — full movement in any direction. This is the most common FTC drivetrain, and almost every team starts from these exact lines (here fwd, str=strafe-right, turn=clockwise):

frontLeft = fwd + str + turn
backLeft = fwd − str + turn
frontRight = fwd − str − turn
backRight = fwd + str − turn
✳️
The magic of the signs. To strafe right, the front-left and back-right wheels roll forward while front-right and back-left roll backward — the diagonal rollers then push the whole robot sideways. Watch the wheel arrows form that "X" pattern in the demo when you strafe.

③ Swerve drive

The Ferrari of drivetrains (mostly FRC). Each corner is a module that can both steer to any angle and drive at any speed. Each module's job is the chassis's overall motion plus the spin contribution at that corner. For a module at position (xᵢ, yᵢ) from the robot's center (forward = +x, left = +y), with chassis velocity (vx, vy) and turn rate ω:

module vx = vx − ω · yᵢ
module vy = vy + ω · xᵢ

speed = √(vx² + vy²)    angle = atan2(vy, vx)

That "+ ω × position" piece is just the fact that during a turn, the corners of the robot whip around faster than the center — and each in a different direction. Each module steers to its own angle and drives at its own speed.

Live demo Forward · strafe · turn → wheels
Sideways motion.
FTC the mecanum teleop almost every team copies first
double y  = -gamepad1.left_stick_y;   // forward (stick up reads negative)
double x  =  gamepad1.left_stick_x;   // strafe right
double rx =  gamepad1.right_stick_x;  // turn clockwise

// Scale all four down together so none exceeds 1.0 (keeps the direction true):
double denom = Math.max(Math.abs(y) + Math.abs(x) + Math.abs(rx), 1.0);

frontLeft .setPower((y + x + rx) / denom);
backLeft  .setPower((y - x + rx) / denom);
frontRight.setPower((y - x - rx) / denom);
backRight .setPower((y + x - rx) / denom);
FRC · WPILib let the kinematics class do the algebra
// SWERVE — give module corner locations once (metres, +x fwd, +y left):
var kinematics = new SwerveDriveKinematics(
        new Translation2d( 0.3,  0.3),   // front-left
        new Translation2d( 0.3, -0.3),   // front-right
        new Translation2d(-0.3,  0.3),   // back-left
        new Translation2d(-0.3, -0.3));  // back-right

ChassisSpeeds speeds = new ChassisSpeeds(vx, vy, omega);  // m/s, m/s, rad/s
SwerveModuleState[] states = kinematics.toSwerveModuleStates(speeds);
for (int i = 0; i < 4; i++) {
    modules[i].setAngle(states[i].angle);              // steer
    modules[i].setSpeed(states[i].speedMetersPerSecond); // drive
}

// TANK — same idea, simpler:
var tank = new DifferentialDriveKinematics(trackWidthMetres);
var w = tank.toWheelSpeeds(new ChassisSpeeds(vx, 0, omega));
// w.leftMetersPerSecond, w.rightMetersPerSecond
🧭
Field-centric driving. Pros usually rotate the joystick command by the robot's heading first, so "push up" always means "away from the driver" no matter which way the robot faces. It's one rotation added in front of the kinematics above — and it needs a good heading estimate, which is exactly what odometry gives us next.

Now that we can command motion, the natural question is: where did the robot actually end up? That's odometry.
