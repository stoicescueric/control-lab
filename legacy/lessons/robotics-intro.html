The loop that runs your whole robot

A robot doesn't "do" one thing — it runs the same loop dozens of times a second, forever. Read sensors, clean them up, decide, act. Repeat. This is exactly the closed loop from Lesson 1, just with names you'll see in real code.

FRC · WPILib the loop is a method that's called for you
public class Robot extends TimedRobot {
    // robotPeriodic() runs automatically every 0.02 s (20 ms).
    @Override
    public void robotPeriodic() {
        double measured = encoder.getRate();          // 1. SENSE
        double clean    = filter.calculate(measured);  // 2. ESTIMATE (Lesson 3)
        double output   = pid.calculate(clean, target);// 3. DECIDE  (Lesson 8)
        motor.setVoltage(output);                       // 4. ACT
    }
}
FTC · SDK you write the loop yourself
waitForStart();
ElapsedTime timer = new ElapsedTime();
while (opModeIsActive()) {
    double dt = timer.seconds();  timer.reset();   // how long since last loop?

    double measured = encoder.getVelocity();        // 1. SENSE
    double clean    = filter.update(measured);       // 2. ESTIMATE
    double output   = controller.update(target, clean, dt); // 3. DECIDE
    motor.setPower(output);                          // 4. ACT
}
⏱️
Why dt matters. Integrals and derivatives (the I and D of PID, every filter, all odometry) are really "per second" math. If your loop runs at a different speed than you assumed, the numbers come out wrong. FRC hands you a fixed 20 ms; in FTC you should measure dt each loop, like above.

The one piece of math every robot needs: ticks → metres

Motors don't know about metres. They have an encoder that counts little ticks as the shaft turns. Your very first job on any mechanism is converting those ticks into something physical — metres travelled, or degrees an arm has rotated. Get this wrong and nothing else (odometry, profiles, paths) can possibly work.

ticks per wheel turn = encoder counts/rev × gear ratio

distance = ticks ÷ (ticks per wheel turn) × π · wheel diameter

The middle part — π · diameter — is just the wheel's circumference: how far it rolls in one full turn. Spin the wheel N revolutions and you've rolled N circumferences. That's the whole idea.

FTC a tiny helper you'll reuse all season
// goBILDA 5202/5203 312-RPM motor as an example:
static final double TICKS_PER_REV  = 537.7;   // counts per output-shaft turn
static final double GEAR_RATIO     = 1.0;     // any EXTRA gearing after the motor
static final double WHEEL_DIAMETER = 0.096;   // metres  (96 mm wheel)

double ticksToMetres(int ticks) {
    double revolutions = ticks / (TICKS_PER_REV * GEAR_RATIO);
    return revolutions * Math.PI * WHEEL_DIAMETER;
}

// later, in the loop:
double metres = ticksToMetres(motor.getCurrentPosition());
FRC · WPILib the library does it once, then hands you metres
// Tell the encoder how far one tick is, then just read .getDistance():
encoder.setDistancePerPulse(Math.PI * kWheelDiameter / kCountsPerRev);

double metres        = encoder.getDistance();  // already in metres
double metresPerSec  = encoder.getRate();      // already in m/s

Try it: an encoder explorer

Spin a motor and watch ticks become distance. Change the encoder resolution, the gearing, and the wheel size, and notice how the same number of ticks can mean wildly different real distances. This is the bug that eats a thousand rookie autonomous routines.

Experiment: set a speed and let it run for a few seconds. Now halve the wheel diameter — the tick count is identical, but the distance is half. A robot told to "drive 2000 ticks" goes a completely different distance on different wheels. That's why we always convert to metres first.
