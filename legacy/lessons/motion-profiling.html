Motion Profiling

Don't tell a mechanism "be there now." Hand it a smooth schedule it can actually follow.

Imagine you want a linear slide to go up 1 metre. The naive way is to set the target to 1 m and let a PID slam it there. The slide lurches, the wheels slip, the whole robot rocks, and it overshoots. The problem: you asked for an instant jump, which needs infinite speed and force. No motor can do that.

A motion profile fixes this. Instead of one far-away target, it produces a moving target — a position for every instant between here and there — that never asks for more speed or acceleration than the mechanism can deliver. The controller chases a target that's always within reach, so the motion comes out smooth, fast, and repeatable.

🚦 A good bus driver. They don't floor it then slam the brakes. They accelerate smoothly to cruising speed, cruise, then ease off early to stop gently. Passengers barely notice. That speed-vs-time story — ramp up, hold, ramp down — is exactly a trapezoidal motion profile.

The trapezoidal profile

The most common profile has three phases. Plot the velocity over time and you get the shape it's named after — a trapezoid:

The math (it's just areas under the line)

Velocity is the slope of position; the distance covered is the area under the velocity graph. With a top speed vmax and acceleration limit amax, for a move of distance d:

time to reach v max: t_ramp = vmax / amax
distance used ramping (×2 ends): d_ramp = vmax² / amax

if d ≥ d_ramp → full trapezoid (there's room to cruise):
  cruise time = (d − vmax²/amax) / vmax

if d < d_ramp → triangle (too short to reach vmax):
  peak speed = √(amax · d)
📐
The triangle case matters. For short moves the mechanism never reaches top speed — it ramps up and immediately ramps back down. Good profile code checks for this. The demo below flips between trapezoid and triangle automatically as you shrink the distance; watch the shape change.

Why this is the perfect partner for feedforward

A profile hands you, at every instant, a target position, velocity, and acceleration. That's a gift: feed the velocity and acceleration straight into the kV and kA feedforward from last lesson, and use PID only to fix the tiny position error. This is how top FRC mechanisms move — fast and dead accurate.

voltage = kV·v_profile + kA·a_profile + PID(pos, pos_profile)
FRC · WPILib TrapezoidProfile + feedforward
var limits = new TrapezoidProfile.Constraints(kMaxVel, kMaxAccel);  // m/s, m/s^2
var profile = new TrapezoidProfile(limits);

var goal     = new TrapezoidProfile.State(targetMetres, 0);  // arrive stopped
var setpoint = new TrapezoidProfile.State(0, 0);             // start here, stopped

@Override
public void robotPeriodic() {
    // Advance the moving setpoint by one 20 ms loop:
    setpoint = profile.calculate(0.02, setpoint, goal);

    double ff = feedforward.calculate(setpoint.velocity);          // kV*v + kA*a
    double fb = pid.calculate(encoder.getDistance(), setpoint.position);
    motor.setVoltage(ff + fb);
}
FRC · WPILib the all-in-one shortcut
// ProfiledPIDController builds the profile internally — just give it the goal:
var controller = new ProfiledPIDController(
        kP, kI, kD, new TrapezoidProfile.Constraints(kMaxVel, kMaxAccel));

double fb = controller.calculate(encoder.getDistance(), targetMetres);
double ff = feedforward.calculate(controller.getSetpoint().velocity);
motor.setVoltage(fb + ff);

Going further: a trapezoid still has sharp corners in acceleration (it jumps from 0 to +a instantly). Limiting how fast acceleration itself can change gives an even gentler S-curve (jerk-limited) profile. Same idea, one more layer of smoothness — great for tall, tippy mechanisms.
