Autonomous routines need the robot's **pose**: position $(x, y)$ and heading $\theta$. No sensor
reports this directly, so it is reconstructed every loop from incremental motion. Doing that
reconstruction accurately — and knowing precisely how it fails — is the core of field-relative
autonomous.

<Problem title="No sensor reports absolute pose, and encoders drift">

There is no "you are here" sensor on an FTC field. Pose must be *integrated* from wheel motion, and
integration has a fundamental weakness: **errors accumulate and never cancel.** Worse, a powered
drive wheel that slips during a hard acceleration or turn reports rotation that did not translate
into ground movement, so its encoder lies. Over a match, the estimate spirals away from reality.
This is the exact limitation that motivates the filters later in the curriculum.

<Analogy>
<strong>Navigating with your eyes closed.</strong> Take ten careful steps while turning and you can
roughly track your position from memory — but small per-step errors compound, and you end up lost.
Odometry is that process, with the same failure mode.
</Analogy>

</Problem>

<Theory title="Integrating motion as a sequence of small arcs">

Each loop, read how far the left and right sides rolled, $d_L$ and $d_R$ (in metres). The forward
step and heading change are:

$$
ds = \frac{d_L + d_R}{2}, \qquad d\theta = \frac{d_R - d_L}{T}
$$

with $T$ the track width. Assuming the step was *straight* ($x \mathrel{+}= ds\cos\theta$) cuts
corners on curves. The accurate model treats each step as a small constant-curvature **arc** — the
*pose exponential* — which matters over a real trajectory:

<JavaCode league="FTC" label="Constant-curvature pose update (the algorithm)">{`double dL = leftPos  - lastLeft;       // metres rolled this loop
double dR = rightPos - lastRight;
lastLeft = leftPos;  lastRight = rightPos;

double ds  = (dL + dR) / 2.0;
double dth = (dR - dL) / TRACK_WIDTH;

double sinC, cosC;
if (Math.abs(dth) < 1e-9) {             // nearly straight -> avoid divide-by-zero
    sinC = 1.0;            cosC = 0.5 * dth;
} else {
    sinC = Math.sin(dth) / dth;
    cosC = (1 - Math.cos(dth)) / dth;
}
x       += ds * (sinC * Math.cos(heading) - cosC * Math.sin(heading));
y       += ds * (sinC * Math.sin(heading) + cosC * Math.cos(heading));
heading += dth;`}</JavaCode>

Two engineering choices sharpen this in practice. **Heading from the IMU:** encoder-derived
$d\theta$ is corrupted by wheel scrub, so most teams take heading from the gyro and use encoders
only for distance. **Dead wheels:** unpowered omni "odometry pods" on light springs measure true
ground motion, immune to drive slip — two parallel pods give forward distance and heading, a third
(perpendicular) captures strafe.

<Callout type="warn" title="Odometry always decays.">
Errors only accumulate. The remedy is periodic correction from an <em>absolute</em> reference — an
AprilTag, a wall-facing distance sensor, field markings. Fusing that correction <em>optimally</em>
with the drifting estimate is exactly what a <strong>Kalman filter</strong> does.
</Callout>

</Theory>

<Deploy title="Read the maintained pose; reset from vision">

In practice a library maintains the pose from dead-wheel pods every loop. Your responsibilities are
tuning its constants (ticks-per-rev, wheel radius, pod offsets) and resetting from an AprilTag when
one is visible.

<JavaCode league="FTC" label="Road Runner pose, corrected by AprilTag">{`drive.updatePoseEstimate();              // integrate one loop of motion
Pose2d pose = drive.localizer.getPose();

double x = pose.position.x;              // configured units (e.g. inches)
double y = pose.position.y;
double heading = pose.heading.toDouble();

// When the camera resolves a tag, trust it and erase the accumulated drift:
if (tag != null) drive.localizer.setPose(tag.fieldPose);`}</JavaCode>

With a reliable pose, the final piece of autonomous is **following a planned path** — the subject of
the next page.
