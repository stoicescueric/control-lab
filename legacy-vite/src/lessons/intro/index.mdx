Control engineering is the discipline of making a physical system track a desired state despite
disturbances. You have a **target** (an arm at 90°, a flywheel at 3000 RPM, the robot parked on a
line) and a single **actuator** (motor power). The entire field answers one question: *how should
the command be computed, every loop, so the system reaches the target and holds it while the world
pushes back?*

<Problem title="A fixed command cannot reject disturbances">

The naïve approach is **open loop**: choose a command once and apply it blindly. Run the arm motor
at 40 % power and hope it lands at 90°. This fails the moment reality deviates from the assumption —
a sagging battery, a defender's bump, friction that changes with wear. The controller never learns
it missed, because it never looks.

<Analogy>
<strong>Microwave vs. shower.</strong> A microwave is open loop — it runs 30 s regardless of how
hot the food gets. A shower is closed loop — you feel the water, compare it to "comfortable," and
adjust the tap continuously. Only one of these survives a cold front.
</Analogy>

The fix is **feedback**: measure the result, compute how far off you are, and correct. That single
idea — the closed loop — is what makes a machine robust.

</Problem>

<Theory title="The closed loop, and the error that drives it">

Five terms describe every feedback system. Learn them once; they appear on every page.

| Term | Definition | On an FTC arm |
|---|---|---|
| **Setpoint** | the target value | "hold 90°" |
| **Measurement** | the current sensor reading | encoder reports 84° |
| **Error** | setpoint − measurement | 6° low |
| **Controller** | the rule mapping error → command | your control code |
| **Actuator** | the element that acts on the world | the motor |

The **error** is the quantity every controller consumes:

$$
e = \text{setpoint} - \text{measurement}
$$

The controller computes a command from $e$, the actuator changes the system, a sensor measures the
result, and the result feeds back to recompute $e$ — continuously.

The demo below makes the difference concrete. Two rooms heat toward **21 °C**. The
<span style={{ color: "#ff6f9c" }}>**open-loop**</span> room holds a fixed heater power; the
<span style={{ color: "#ffc24d" }}>**closed-loop**</span> room measures its temperature and adjusts.
Drag the outdoor temperature down and observe which one holds setpoint.

<Heater />

<Callout type="idea" title="The takeaway:">
the closed-loop room holds 21 °C regardless of the disturbance — its error grows, so its command
grows to match. The open-loop room cannot detect that it is failing. This is precisely why feedback
is non-negotiable on a competition robot.
</Callout>

<Deploy title="The control-loop skeleton on a Control Hub">

Every controller in this curriculum slots into the same periodic loop. The structure never changes
— only the line that computes the command.

<JavaCode league="FTC" label="The shape of every feedback loop">{`while (opModeIsActive()) {
    double measurement = arm.getCurrentPosition();   // read the sensor
    double error = setpoint - measurement;           // compare to the target

    double command = controller.update(error);       // <-- the only part that changes
    arm.setPower(clamp(command, -1, 1));             // act on the world
}`}</JavaCode>

The rest of this curriculum fills in that one line — but only after two prerequisites. **Filters**
clean the noisy *measurement* so the controller is not misled; **controllers** (PID, feedforward,
motion profiling) define the *rule*. Before either runs correctly on hardware, the continuous maths
has to become a discrete loop in real units — which is the next page.

</Deploy>
